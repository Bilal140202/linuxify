# Launcher Architecture

> **Audience**: AI coding agents implementing the launcher shim layer and `linuxify run` internals, and human contributors debugging "the tool runs inside proot but not from Termux" issues.
>
> **Scope**: This document covers the launcher subsystem — the small shell-script shims in `$PREFIX/bin/` that turn a Termux shell command like `cline` into a proot-bounded execution of the real `cline` binary inside Ubuntu. For the runtime layer that manages Node/Python/Rust versions *inside* a distro, see [runtime-management.md](runtime-management.md). For the bootstrap that creates the distro in the first place, see [../05-bootstrap/bootstrap-design.md](../05-bootstrap/bootstrap-design.md).

## 1. Problem Statement

When a user types `cline` in a Termux shell, six things must happen in the right order, all transparent to the user, before the actual `cline` binary can do its work:

1. The Termux shell resolves `cline` via `$PATH` and finds it in `$PREFIX/bin/cline`.
2. That file is not the real `cline` binary — it is a Linuxify-generated shim. The shim must `exec` into `linuxify run cline -- "$@"`.
3. `linuxify run` must determine which distro `cline` is installed in (the active distro from `state.json`), which runtime version it requires (from `~/.linuxify/packages/cline.json`), and which environment variables it expects (from the package YAML's `env:` block).
4. `linuxify run` must construct a `proot-distro login` invocation that binds the user's working directory, sets up the environment, drops to the `linuxify` user, and execs the actual `cline` binary.
5. Inside the proot, `cline` runs, reading from the user's stdin, writing to the user's stdout/stderr, and responding to the user's Ctrl-C.
6. When `cline` exits, its exit code must propagate back to the Termux shell unchanged.

The launcher subsystem is the shim that does all of this. It is the single most user-visible component of Linuxify — if the launcher is broken, every Linuxify-managed tool is broken. It is also the most performance-sensitive component, because it is in the hot path of every command invocation. A 500 ms launcher overhead would make `cline --help` feel sluggish; the target is 50–100 ms (see §13).

The launcher is *not* a virtualization layer. It does not translate syscalls (proot does that), it does not manage processes (the kernel does that), and it does not provide isolation (proot does not either — see §14). It is a thin, deterministic, fast shim whose only job is to set up the proot invocation correctly and get out of the way.

## 2. Launcher Format

A Linuxify launcher is a small POSIX shell script, generated by `linuxify add` (and re-generated by `linuxify repair launchers`), placed at `$PREFIX/bin/<name>`. The Termux `$PREFIX` is `/data/data/com.termux/files/usr`, which is on the user's `$PATH` by default. Example launcher for `cline`:

```sh
#!/data/data/com.termux/files/usr/bin/sh
# Auto-generated by linuxify 0.1.0. Do not edit.
# Package: cline (v1.2.0) — AI coding agent that runs in your terminal
# Distro:  ubuntu (active)
# Runtime: node 22.11.0
#
# To regenerate: linuxify repair launchers cline
# To uninstall:  linuxify remove cline

LINUXIFY_PKG="cline"
LINUXIFY_DISTRO="ubuntu"
LINUXIFY_RUNTIME="node"
LINUXIFY_RUNTIME_VERSION="22.11.0"

export LINUXIFY_INVOKED_AS="${0##*/}"
export LINUXIFY_LAUNCHER="$0"

exec linuxify run "$LINUXIFY_PKG" -- "$@"
```

The shebang is absolute and points at Termux's `sh` (which is `dash` on modern Termux, not bash). This is deliberate: the launcher must work even if the user's interactive shell is `zsh` or `fish`, and `dash` has the fastest startup of any POSIX shell on Android.

The body of the launcher is intentionally minimal. It sets four environment variables (used by `linuxify run` for diagnostics and by the package's own code if it cares to look itself up), then `exec`s into `linuxify run`. The `exec` is critical: it replaces the shell process with the `linuxify` process, so the process tree stays flat and signal propagation works naturally (see §8).

The `"$@"` at the end forwards all arguments verbatim, including those with spaces, quotes, or special characters. This is the standard POSIX idiom for argument forwarding and is preserved exactly.

## 3. Why Shell Scripts, Not Symlinks

A reasonable question is why the launcher is a shell script rather than a symlink to `~/.linuxify/runtimes/ubuntu/node/22.11.0/bin/cline` (the real binary inside the proot). The answer has three parts.

First, **a symlink would bypass Linuxify's runtime hooks**. The whole point of `linuxify run` is that it can do preflight checks (is the distro installed? is the package installed? is the runtime version correct?), set up the environment (merge Termux env with proot env with package env), and bind-mount the user's working directory. A symlink would skip all of this and the binary would either fail to find its libraries (proot is not entered) or, worse, run with the wrong environment and produce confusing bugs.

Second, **a symlink would not enter the proot**. The real `cline` binary lives inside `~/.linuxify/distros/ubuntu/home/linuxify/.local/share/linuxify/runtimes/node/22.11.0/lib/node_modules/cline/dist/cli.js`. Even if Termux could resolve a symlink into that path, the binary would still need proot to translate syscalls (because it expects `/proc`, `/sys`, `/dev` to look like Linux, not Android). Entering proot is non-negotiable; the question is only *how* the entry is triggered. A shell script that calls `linuxify run` is the cleanest answer.

Third, **shell scripts are debuggable and inspectable**. A user who suspects the launcher is misbehaving can `cat $PREFIX/bin/cline` and see exactly what it does. A symlink is opaque. The transparency is worth the few milliseconds of shell-startup cost.

The exception is the "Direct" launcher variant (see §11), which is a symlink-like optimization for performance-critical trusted tools where the runtime hook overhead is unacceptable.

## 4. `linuxify run` Internals

When invoked as `linuxify run cline -- --help`, the `linuxify run` subcommand performs an eight-step dance. Each step is documented below; the corresponding code is in `launcher/run.ts`.

**Step 1: Look up package metadata.** Read `~/.linuxify/packages/cline.json`. This file is written by `linuxify add` and contains the package's name, version, runtime, runtime_min_version, install path inside the distro, env block, and the path to the actual binary. If the file is missing, `linuxify run` errors with "package 'cline' is not installed; try `linuxify add cline`" and exits non-zero.

**Step 2: Look up active distro.** Read `~/.linuxify/state.json`'s `active_distro` field. This is the distro the package was installed in (or, after a `linuxify use` switch, the distro the user is currently targeting). If `state.json` is missing or `active_distro` is unset, error with "no active distro; run `linuxify init` first" and exit.

**Step 3: Verify distro is installed.** Check that `~/.linuxify/distros/<active>/` exists and contains a marker file `.linuxify-distro-installed`. If not, error with "distro '<active>' is not installed; run `linuxify distros install <active>` or `linuxify init`" and exit. This check is fast (a single `stat` call) and catches the common case where the user has deleted the distro directory manually.

**Step 4: Verify package is installed in that distro.** Check that the package's install path inside the distro exists. For `cline` installed via npm global, this is `~/.linuxify/distros/<active>/home/linuxify/.linuxify/runtimes/node/22.11.0/lib/node_modules/cline/`. If the path is missing (e.g. the user uninstalled cline via `npm uninstall -g` inside the proot without telling Linuxify), error with "package 'cline' is not installed in distro '<active>'; run `linuxify add cline`" and exit.

**Step 5: Build environment.** Three environment layers are merged, in order of increasing precedence:

- **Termux env**: the environment variables currently set in the `linuxify run` process (inherited from the Termux shell). This is the base layer.
- **proot env**: variables Linuxify always sets inside the proot: `HOME=/home/linuxify`, `USER=linuxify`, `LOGNAME=linuxify`, `SHELL=/bin/bash`, `TERM=$TERM`, `LANG=en_US.UTF-8` (or the configured locale), `PATH=<runtime bin>:/usr/local/bin:/usr/bin:/bin`. The full table is in §5.
- **Package env**: variables from the package YAML's `env:` block. For `cline`, this is `CLINE_PLATFORM=linux` and `FORCE_COLOR=1`. These override proot env, which overrides Termux env.

**Step 6: Construct proot invocation.** The actual command looks like:

```sh
exec proot-distro login \
  --distro ubuntu \
  --user linuxify \
  --bind /sdcard \
  --bind /sdcard:/host-sdcard \
  --bind "$PWD:/workspace" \
  --termux-home \
  --env CLINE_PLATFORM=linux \
  --env FORCE_COLOR=1 \
  --env LINUXIFY_PKG=cline \
  -- /home/linuxify/.linuxify/runtimes/node/22.11.0/bin/node \
    /home/linuxify/.linuxify/runtimes/node/22.11.0/lib/node_modules/cline/dist/cli.js \
    --help
```

The `--termux-home` flag (a proot-distro feature) binds the Termux home directory into the proot at `/home/linuxify/host`, so the user's Termux files are accessible. The `--bind "$PWD:/workspace"` makes the user's current working directory available at `/workspace` inside the proot, and the proot's startup script `cd`s to `/workspace` before execing the binary.

**Step 7: exec proot.** The `exec` call replaces the `linuxify run` process with `proot`. From this point on, Linuxify is no longer in the loop; the kernel, proot, and the package binary handle everything. Signal propagation works naturally because proot is the parent process of the binary, and proot forwards signals it receives (see §8).

**Step 8: Capture exit code.** Because `linuxify run` `exec`s proot, it never returns to capture the exit code itself. Instead, the wrapper shell script (or, in the case of `linuxify run` invoked directly, the parent shell) sees proot's exit code, which is the package binary's exit code. For logging purposes (§4 step 1 in the original eight-step list — the run log), Linuxify uses a `ptrace`-based wrapper in `launcher/run.ts` that intercepts the `wait()` syscall to record the exit code in `~/.linuxify/logs/runs/cline-<timestamp>.log`. This wrapper is optional and disabled by default for performance; it can be enabled per-package via `config.toml`.

## 5. Environment Propagation

The environment inside the proot is a careful merge of Termux-side and proot-side variables. The table below enumerates every variable Linuxify knows about.

| Variable | Source | Treatment |
|---|---|---|
| `PATH` | rewritten | `/home/linuxify/.local/share/linuxify/runtimes/<active>/<ver>/bin` first, then `/usr/local/bin:/usr/bin:/bin`. Termux's `$PREFIX/bin` is *not* included (it would expose Termux binaries inside the proot and confuse `which`). |
| `HOME` | rewritten | `/home/linuxify` inside the proot. The user's Termux home is bound at `/home/linuxify/host`. |
| `USER`, `LOGNAME` | rewritten | `linuxify` |
| `SHELL` | rewritten | `/bin/bash` (or the distro's default) |
| `TERM` | passthrough | Preserved so color and cursor control work. |
| `COLORTERM` | passthrough | Preserved for truecolor support. |
| `LANG`, `LC_ALL` | rewritten | The locale configured in `config.toml` (default `en_US.UTF-8`). |
| `TZ` | passthrough | So the package sees the user's local timezone. |
| `SSH_AUTH_SOCK` | passthrough | Bound via `--bind` so SSH agent forwarding works inside the proot. |
| `SSH_TTY`, `SSH_CONNECTION` | passthrough | For SSH-into-Termux sessions. |
| `LINUXIFY_PKG` | set | The package name being run. |
| `LINUXIFY_DISTRO` | set | The distro name. |
| `LINUXIFY_RUNTIME` | set | The runtime name (e.g. `node`). |
| `LINUXIFY_RUNTIME_VERSION` | set | The runtime version (e.g. `22.11.0`). |
| `LINUXIFY_INVOKED_AS` | set | The name the user typed (e.g. `cline`). |
| Package-defined `env:` block | set | Per the package YAML. Last word — overrides anything above. |
| `ANDROID_ROOT`, `ANDROID_DATA` | stripped | Android-specific; their presence inside the proot would confuse some CLIs that detect Android. |
| `BOOTCLASSPATH`, `CLASSPATH` (Android's) | stripped | Same reason. |
| `RANDOM`, `OLDPWD` | stripped | Shell internals that would be wrong inside the proot. |

The stripping of Android-specific variables is what makes `process.platform === "linux"` actually true inside the proot, which is the whole point of Linuxify from a CLI's perspective.

## 6. Working Directory

The user's current working directory in Termux is preserved by binding it into the proot at `/workspace` and `cd`-ing there before execing the binary. A user who runs `cd /sdcard/project && cline` will find that `cline` sees `/sdcard/project` (or, more precisely, `/workspace` inside the proot, which is the same files via the bind) as its working directory.

The `/workspace` alias is intentional rather than binding at the original path. Two reasons: (1) the original path might be `/sdcard/project`, which is a real Android path; binding it at the same path inside the proot is technically possible but conflicts with the proot's `/sdcard` bind; (2) using a fixed alias lets package authors write code that references `/workspace` knowing it will always be the user's CWD, regardless of where they invoked the launcher from. This convention is documented in [../10-plugin-sdk/extension-api.md](../10-plugin-sdk/extension-api.md).

If the user's CWD is not bindable (e.g. it's a path under `/proc` or some other kernel-only path), Linuxify falls back to `cd /home/linuxify/host` (the Termux home bind) and prints a warning. This is rare in practice but the doctor surfaces it.

## 7. Bind Mounts

Default binds applied to every `linuxify run` invocation:

- `/sdcard` → `/sdcard` (so the package can read/write the user's external storage, which is where most users keep their projects).
- `/sdcard` → `/host-sdcard` (alias for compatibility with package code that looks for `/host-sdcard`).
- `$HOME` (Termux home) → `/home/linuxify/host` (so the package can read the user's Termux dotfiles, e.g. `.gitconfig`).
- `$PWD` → `/workspace` (the user's CWD; see §6).
- `/tmp` → `/tmp` (Termux's `/tmp` is bindable; this lets the package write temp files that are visible from Termux if needed).
- `$SSH_AUTH_SOCK` → `$SSH_AUTH_SOCK` (if set, so SSH agent forwarding works).

Additional binds can be requested via the `--bind` flag to `linuxify run`:

```sh
linuxify run cline --bind /data/local/tmp --bind /system/etc/hosts
```

Or, more durably, via the package YAML's `binds:` block, which is applied to every invocation of that package:

```yaml
# packages/cline.yml
binds:
  - /data/local/tmp
  - /system/etc/hosts:/etc/hosts
```

Binds are a security-relevant surface (see §14); the launcher applies them only after verifying they appear in the user's `config.toml` `allowed_binds` list or in the package YAML (which is itself trusted because it was installed via `linuxify add`).

## 8. Signal Handling

The launcher `exec`s proot, which `exec`s the package binary. The process tree is therefore:

```
bash (Termux shell)
└── cline (launcher shell script, exec'd into)
    └── linuxify run (exec'd into)
        └── proot-distro (exec'd into)
            └── proot
                └── node (cline binary)
                    └── ... (cline's child processes)
```

Because every step is an `exec`, the process tree is actually flat — there is no intermediate shell lingering. When the user presses Ctrl-C in the Termux shell, the kernel sends SIGINT to the foreground process group, which includes `node`. The signal arrives at `node` directly, with no launcher code in the way.

The same is true for SIGTERM (from `kill <pid>`), SIGQUIT (Ctrl-\), SIGHUP (terminal close), SIGSTOP (Ctrl-Z), and SIGCONT (fg). proot itself does not intercept these signals; it forwards them transparently.

One subtlety: SIGKILL cannot be caught or forwarded, and killing the launcher's PID before the `exec` to proot completes (a window of ~50 ms) will leave a half-started proot. The launcher does not attempt to handle this; the kernel cleans up the proot's child processes when their parent dies. In practice, this is not a problem because the launcher execs as fast as possible.

## 9. TTY & PTY

A TTY-aware CLI (e.g. `vim`, `top`, `htop`, `cline`'s interactive prompt) needs to know it is connected to a TTY, or it will fall back to non-interactive behavior. Linuxify's launcher detects whether stdin is a TTY (`isatty(0)`) and passes the file descriptor through to proot unchanged. proot, in turn, passes it through to the binary. If the user is running `cline` interactively in Termux, the binary sees a TTY and behaves interactively.

The complications arise in three edge cases:

- **Piped input**: `echo "hello" | cline` — stdin is a pipe, not a TTY. The launcher detects this and does not allocate a PTY. The binary sees a pipe and reads from it normally. This works for all CLIs.
- **PTY-required binaries**: a small number of binaries (notably `ssh` asking for a password, and some TUI libraries) require a PTY even when stdin is a pipe. These fail with "not a tty" or hang silently. The workaround is `linuxify run --pty cline`, which allocates a PTY via `openpty(3)` and bridges it to the binary's stdin/stdout. This is opt-in because PTY allocation costs ~10 ms and changes buffering semantics.
- **Line buffering vs char buffering**: when stdin is a pipe, the binary's stdout defaults to line-buffered (or fully buffered, depending on the binary). When stdin is a TTY, stdout is char-buffered. Some CLIs behave differently in these two modes (e.g. they suppress color output when stdout is not a TTY). Linuxify does not change buffering; it passes the file descriptors through unchanged. The `FORCE_COLOR=1` env var (set by many package YAMLs) is the standard workaround for binaries that check `isatty(stdout)`.

For the rare CLI that needs a PTY but is invoked non-interactively, the doctor's `pty_check` (see [../07-doctor/diagnostics.md](../07-doctor/diagnostics.md)) detects the failure mode and suggests `--pty`.

## 10. Long-running Processes

A long-running process started via the launcher (e.g. `linuxify run cline serve` to start a long-lived agent server) lives as long as the Termux session that launched it. When the user closes Termux (swipes it away in the recent-apps view, or the Android OOM killer reclaims it), all processes in the Termux session — including the proot and the binary inside it — are killed.

This is a fundamental Termux limitation, not a Linuxify one. The mitigations are:

- **Termux:boot** (a Termux add-on): install it, add a `~/.termux/boot/` script that starts the long-running process at device boot. The process runs in a Termux session that survives device reboot.
- **`nohup`**: `nohup cline serve &` detaches the process from the controlling terminal, so closing Termux does not kill it. The process is still killed when Termux itself is killed, but it survives the user closing the foreground session.
- **`tmux`** (or `screen`): run a tmux server inside the proot, start the long-running process inside a tmux session, and detach. The tmux server runs as long as the proot session is alive; the user can reattach with `tmux attach` from a new Termux session. This is the most robust option for genuinely long-lived processes.

Linuxify documents these workarounds in `linuxify help long-running` but does not implement its own daemonization mechanism. Daemon management is out of scope for v1 (see [../15-roadmap/release-roadmap.md](../15-roadmap/release-roadmap.md) for v2 plans).

## 11. Launcher Variants

Three launcher variants are supported, selected per-package via the package YAML's `launcher:` field.

### Standard (default)

The shell-script shim described in §2. Used by 95% of packages. Adds ~50–100 ms overhead per invocation (mostly proot startup, not the shell script itself — the shell script is ~5 ms). Recommended for any package that benefits from Linuxify's runtime hooks: env setup, working directory binding, run logging.

### Direct

A symlink directly to the binary inside the distro, bypassing `linuxify run` entirely. Used for performance-critical trusted tools where the runtime hook overhead is unacceptable. Example: a package that the user invokes hundreds of times per minute (e.g. a code formatter invoked on every save). The symlink is created at `$PREFIX/bin/<name>` and points at the absolute path of the binary inside the distro.

Direct launchers lose: env merging (the package's `env:` block is not applied), CWD binding (`/workspace` is not set up — though the user's CWD is still accessible via the `/sdcard` bind), run logging, and the preflight checks. They are also distro-specific: if the user switches distros, direct launchers point at the wrong distro's binary and must be regenerated with `linuxify repair launchers`. For this reason, direct launchers are opt-in and the package YAML must declare `launcher: direct` explicitly.

### Custom

A user-provided launcher script at `~/.linuxify/launchers/<name>.sh`. Linuxify symlinks `$PREFIX/bin/<name>` to this script. Used for exotic cases: a package that needs a non-standard proot invocation, a wrapper that does pre/post hooks, a launcher that targets a different distro than the active one. The script must `exec linuxify run` (or `exec proot-distro login` directly) at the end; otherwise the binary will not run inside the proot. Custom launchers are entirely the user's responsibility and are not validated beyond a syntax check.

## 12. Launcher Regeneration

`linuxify repair launchers` regenerates every launcher in `$PREFIX/bin/` from the current package YAMLs and the current `state.json`. It is idempotent: a launcher whose content matches what Linuxify would generate is left untouched (compared by SHA-256 of the rendered template). The command is invoked automatically by:

- `linuxify use <distro>` (because the active distro change might change the launcher's `LINUXIFY_DISTRO` variable, even if the package itself is not affected).
- `linuxify upgrade <pkg>` (because the runtime version might have changed).
- `linuxify self-update` (because the launcher template might have changed in the new Linuxify version).

It can also be invoked manually by the user, e.g. after editing `config.toml`'s `[run]` section or after a manual `npm install -g <pkg>` inside the proot (which would change the binary path the launcher points at).

`linuxify repair launchers <pkg>` regenerates only one package's launcher. `linuxify repair launchers --dry-run` shows what would change without writing anything.

## 13. Performance

The launcher overhead target is **≤100 ms** per invocation, with a soft target of 50 ms. The breakdown on a Snapdragon 7-series phone:

| Step | Time |
|---|---|
| Shell script parse and exec | 5 ms |
| `linuxify run` TypeScript startup (Node cold start) | 30 ms |
| Preflight checks (Steps 1–4 of §4) | 5 ms |
| Environment construction (Step 5) | 2 ms |
| `proot-distro login` startup | 40 ms |
| Binary cold start (Node + cline) | 200–500 ms (not counted as launcher overhead) |
| **Launcher overhead total** | **~80 ms** |

The dominant cost is `proot-distro login` itself, which is the cost of proot's syscall translation setup. Linuxify does not control this; it is a property of proot. The "Direct" launcher variant (§11) bypasses the launcher overhead entirely for trusted tools, at the cost of losing the runtime hooks.

Mitigations under exploration for v2:

- **Persistent proot session**: keep a proot process running in the background and `exec` into it via a Unix socket, avoiding the 40 ms proot startup. This is non-trivial because proot is not designed for this; it would require an LD_PRELOAD shim.
- **Linuxify daemon**: a long-running `linuxifyd` process that caches the package metadata and distro state, reducing the 30 ms TypeScript startup. Would add complexity and a new failure mode (the daemon can crash); deferred to v2.
- **Rewrite launcher in C**: replace the shell script with a small C binary that does the same thing in <1 ms. This is a marginal optimization (saves ~5 ms) and the maintainability cost is high; not currently planned.

## 14. Security

The launcher runs with the user's Termux privileges. It is not setuid, does not invoke `sudo`, and does not request any Android permissions beyond what Termux already has. proot itself does not escalate privileges — it is a user-space syscall translator that cannot grant capabilities the calling process does not have.

The important caveat is that **proot is not a security boundary**. A process inside the proot can escape the bind-mount sandbox by directly invoking syscalls that proot does not intercept (proot intercepts only the syscalls needed to fake the filesystem; it does not intercept `clone`, `socket`, or most other syscalls). A malicious package run inside the proot can therefore read any file the Termux user can read, open network connections, and spawn processes outside the proot.

This is the same trust model as Termux itself: the user installs software at their own risk. Linuxify does notarize built-in package manifests (see [../09-registry/registry-format.md](../09-registry/registry-format.md)) and warns loudly when installing custom or unsigned packages, but it does not sandbox untrusted code. For users who need to run untrusted CLIs, the recommendation is to do so inside a separate Termux installation (Android supports multiple Termux profiles via the `com.termux.window` add-on) or in an emulator.

The doctor's `security_audit` check (see [../13-security/security-model.md](../13-security/security-model.md)) scans installed packages for known-bad versions and reports any findings. It is not a substitute for trust, but it catches the common case of "the user installed a package from a typo-squatted URL."
