# ADR-001: Use proot over chroot or QEMU

- Status: accepted
- Date: 2025-01-10
- Deciders: Linuxify core team
- Context: Linuxify needs a mechanism to run a Linux userland (Ubuntu, Debian, Arch, Alpine) inside an Android Termux process so that desktop-developer CLIs (Cline, Codex, Aider, Goose, Gemini CLI, OpenHands, Freebuff) can execute against a glibc-based, x86_64-assuming, `/proc`-and-`/sys`-expecting environment rather than the bionic-libc, `process.platform === "android"`, Android-kernel environment Termux natively provides. The mechanism must satisfy four hard constraints. First, it must require **no root access**, because the vast majority of Android users do not have rooted devices and requiring root would shrink Linuxify's addressable audience by roughly two orders of magnitude. Second, it must support **arbitrary distros** (not just one blessed image) so that users can pick Ubuntu for compatibility, Alpine for size, Arch for freshness, or Debian for stability. Third, it must be **transparent to the running CLI** — the CLI should see `/proc/self/exe`, `uname`, `process.platform`, and `glibc` symbols that match a real Linux system, not Android. Fourth, it must be **lightweight enough for interactive use**: a CLI that takes 5 seconds to start is unusable for the "I run `cline --help` and expect an instant response" workflow.

  Three options were considered. **chroot** requires `CAP_SYS_CHROOT`, which on Android means root; it is the fastest option (no syscall translation) but is unavailable to 99% of users. **QEMU user-mode emulation** runs an entire VM (or, in user-mode, translates syscalls for a foreign architecture); it is the heaviest option, has the worst startup latency, and is overkill for the common case where the host and guest are both aarch64. **proot** is a ptrace-based syscall translator that fakes `chroot` using `LD_PRELOAD` and a userspace filesystem-namespace shim; it requires no root, supports arbitrary rootfs images, and adds 10–40 ms of overhead per syscall-bound operation — noticeable in microbenchmarks but invisible in interactive CLI use. A variant, **proot-distro**, wraps proot with rootfs download, unpack, and lifecycle management, and is the de facto standard inside the Termux community.

- Decision: Linuxify uses **proot** (specifically the `proot-distro` wrapper) as its default and primary distro backend. The DistroProvider interface (see [distro-management](../05-bootstrap/distro-management.md) §1) is pluggable, so a future chroot-based or QEMU-based provider can be added without re-architecting Linuxify, but proot is the only provider shipped in v1 and the only provider the maintainers commit to supporting. The choice is encoded in the bootstrap pipeline's Stage 3 (see [bootstrap-design](../05-bootstrap/bootstrap-design.md) §2), which calls `proot-distro login` to enter the rootfs.

- Consequences:
  - **Positive**: No root required, which preserves Linuxify's "works on any Android 9+ device" promise. Arbitrary distros supported via rootfs tarballs. Transparent to the running CLI (`process.platform` reports `linux` after patching, glibc is the real glibc). Mature ecosystem — `proot-distro` is widely used in Termux, well-documented, and actively maintained. Startup latency of 40–80 ms per `linuxify run` invocation is acceptable for interactive use (see launcher performance budget in [launcher-architecture](../06-launcher/launcher-architecture.md) §13).
  - **Negative**: proot is **not a security boundary** (see [security-model](../13-security/security-model.md) §15) — a malicious CLI inside proot has the same filesystem and network access as the Termux user. Syscall translation adds overhead to syscall-heavy workloads (compilers, databases); a `npm install` that does 10,000 `stat` calls is measurably slower than on bare Linux. proot occasionally segfaults on new Android versions (e.g., Android 14 changed ptrace semantics in a way that broke older proot builds); Linuxify mitigates by pinning a known-good proot version and shipping a doctor check (`E_PROOT_SEGFAULT` pattern in [diagnostics](../07-doctor/diagnostics.md) §3.3). Some syscalls (notably `ptrace` itself, and certain `io_uring` operations) are not translated correctly and fail with `ENOSYS`; the patcher (see [patcher-engine](../08-patcher/patcher-engine.md) §2) handles the common cases.
  - **Neutral**: The pluggable DistroProvider interface means a v2 chroot provider (for rooted users who want speed) or a v3 QEMU provider (for running x86-only macOS tools on ARM) can be added later without breaking existing users. This is documented as a v3+ goal in the [vision extension](../19-future/vision-extension.md).

- Alternatives Considered:
  - **chroot**: rejected because root is required; would cut the user base by ~99%.
  - **QEMU full-system emulation**: rejected for v1 because the performance cost (10–100× slower than native) makes interactive CLI use painful, and because the common case (aarch64 host, aarch64 guest) does not need emulation. Reserved for v3+ as an opt-in for running x86-only tools.
  - **QEMU user-mode emulation**: rejected because it still requires binfmt_misc registration (which needs root on Android) and has the same performance concerns as full-system.
  - **A custom ptrace-based translator written from scratch**: rejected because the engineering cost is enormous, the Termux community already has proot, and reinventing it would diverge from community expectations.
