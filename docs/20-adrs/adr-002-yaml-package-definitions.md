# ADR-002: YAML for package definitions

- Status: accepted
- Date: 2025-01-10
- Deciders: Linuxify core team
- Context: Linuxify needs a serialization format for package definitions — the files that declare how to install, patch, and verify a CLI tool like `cline` or `aider` (see [package-spec](../09-registry/package-spec.md) §2 for the full schema). These files are the primary surface on which the open-source community contributes: a contributor who wants to add support for a new CLI writes one YAML file and opens a PR. The format choice therefore has three constituencies. First, **human authoring**: the format must be readable and writable by hand, because most contributors will edit these files in a text editor, not generate them programmatically. Second, **machine validation**: the Linuxify CLI must be able to load the file, validate it against a schema, and produce a helpful error message if a field is missing or mistyped. Third, **ecosystem familiarity**: the format should be one that the likely contributor pool (developers who use GitHub Actions, Docker Compose, Ansible, Kubernetes) already knows, so that contributing a package definition does not require learning a new syntax.

  Four options were considered. **YAML** is human-readable, supports comments, supports nested arrays and objects, has a mature schema-validation ecosystem (JSON Schema via ajv), and is the format used by GitHub Actions, Docker Compose, Ansible, and Kubernetes — all of which overlap with Linuxify's contributor pool. Its main cost is whitespace sensitivity, which produces confusing errors when a contributor mixes tabs and spaces. **JSON** is universally parseable but hostile to hand-authoring: no comments, strict quoting, trailing commas forbidden, verbose nested structures. **TOML** is human-readable and comment-friendly but its handling of nested arrays-of-objects (which the `patches:` and `doctor:` blocks require) is awkward — you end up with `[[patches]]` and `[[patches.verify]]` sections that are harder to read than YAML's indented blocks. **A custom DSL** (e.g., a TypeScript-flavored configuration language like `typescript` config files, or a Lisp-like syntax) would be more powerful but adds a learning curve that suppresses casual contributions.

- Decision: Linuxify uses **YAML 1.2** for package definitions, validated against a strict JSON Schema (draft 2020-12) at load time via `ajv`. The schema is documented in [package-spec](../09-registry/package-spec.md) §1 and the validation workflow is `linuxify package lint <file>` (see [cli-specification](../03-cli/cli-specification.md) §4). The schema enforces `additionalProperties: false` so that mistyped field names (e.g., `runtime_min_verison` instead of `runtime_min_version`) produce a hard error rather than silently being ignored.

- Consequences:
  - **Positive**: Contributors who have written a GitHub Actions workflow or a `docker-compose.yml` can write a Linuxify package definition with zero learning curve. Comments in the file (`# patches the platform check`) make the file self-documenting. Schema validation catches the most common errors (typos, wrong types, missing required fields) at `linuxify package lint` time, before the file is ever loaded by the runtime. The `cline.yml` example in the [project context](../../.agent-context.md) §6 is typical of the format's expressiveness.
  - **Negative**: Whitespace sensitivity is a real cost — a contributor who pastes a `patches:` block with the wrong indentation gets a confusing parse error. We mitigate with a `linuxify package lint` that produces a column-precise error message ("line 23, column 5: expected 2 spaces, found 4") and with editor plugins (we ship a VS Code schema association that enables inline validation). YAML's alias/anchor feature (`&`/`*`) is powerful but error-prone; we discourage it in package definitions and the linter warns on its use. YAML's "Norway problem" (`no` is parsed as boolean false) means we must quote two-letter country codes in any future i18n metadata — documented in the schema.
  - **Neutral**: The schema is versioned (`schema_version: 1` field). Additive changes (new optional fields) are minor-version bumps; breaking changes are major-version bumps with a 2-version deprecation window (see [package-spec](../09-registry/package-spec.md) §13). The format choice is reversible — a v2 could ship TOML alongside YAML and read both — but no contributor has requested this and we do not anticipate it.

- Alternatives Considered:
  - **JSON**: rejected because no comments and verbose quoting make hand-authoring painful; the contributor experience would be measurably worse.
  - **TOML**: rejected because nested arrays-of-objects (which `patches:` and `doctor:` require) are awkward; the `[[patches]]` syntax is harder to read than YAML's indented blocks for the specific shapes Linuxify needs.
  - **Custom DSL**: rejected because the learning curve would suppress casual contributions, and because the maintenance burden of a custom parser is not justified by the modest expressive gains over YAML.
  - **JSON5 / JSONC**: rejected because they are less widely known than YAML and offer no compelling advantage for our schema shape.
