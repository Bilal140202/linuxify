# ADR-003: TypeScript + Node.js for the CLI core

- Status: accepted
- Date: 2025-01-10
- Deciders: Linuxify core team
- Context: Linuxify is a developer-tooling CLI whose primary job is to install, patch, and launch other developer-tooling CLIs — most of which are themselves Node.js applications (Cline, Codex, Aider's JS tooling) or have first-class Node bindings. The choice of implementation language for the Linuxify CLI core is therefore consequential. Three hard requirements drove the decision. First, the language's runtime must be **installable on Android Termux without root** and must be **the same runtime that the managed CLIs use**, so that Linuxify can inspect, patch, and debug those CLIs using the same toolchain rather than maintaining two parallel ecosystems. Second, the language must have a **mature ecosystem for CLI development** — argument parsing, terminal color/TTY handling, TOML/YAML parsers, child-process management, fs APIs — so that we are not reinventing basic infrastructure. Third, the language must support **static type checking** so that the substantial surface area of the Linuxify core (7 subsystems, 18 subcommands, 9 state files) can be refactored with confidence.

  Three options were considered. **TypeScript on Node.js** matches the runtime of the managed CLIs (Node), has the largest ecosystem for CLI development on Node (commander, yargs, chalk, ora, ajv, zod), has first-class static typing via tsc, and can run inside its own proot Ubuntu using its own Node runtime — eliminating the "two Node versions" problem. The chicken-and-egg bootstrap (the very first `linuxify init` must install Node before Linuxify can run) is solved with a tiny POSIX pre-bootstrap script. **Rust** produces a single self-contained binary with no runtime dependency, has best-in-class performance and memory safety, and has a growing CLI ecosystem (clap, serde, tokio). However, Rust does not run on the Node runtime — meaning Linuxify-the-binary and Linuxify-the-toolchain-manager would be separate stacks, doubling the maintenance surface for any contributor who works on both. Distribution is also more complex (cross-compilation per target triple; signed binaries per platform). **Go** produces a single static binary, has excellent CLI ergonomics (cobra, urfave/cli), and has fast compilation. However, Go's runtime is distinct from Node's, and the CLIs Linuxify manages are overwhelmingly Node or Python — meaning a Go-based Linuxify would be a foreign object in its own ecosystem, and contributors who want to debug a managed CLI would need to learn Go *and* Node.

- Decision: Linuxify is implemented in **TypeScript** (strict mode, ESM, target ES2022) running on **Node.js LTS** (v20+). The CLI is distributed via three channels: `npm install -g linuxify` (the canonical install for users who already have Node), the Termux `pkg install linuxify` package (for users who do not), and GitHub Releases with detached `.sig` signatures (for users who want to verify). The chicken-and-egg bootstrap is handled by a ~200-line POSIX shell pre-bootstrap that installs Node into Termux's `$PREFIX` first, then `npm install -g linuxify`, then hands control to the TypeScript CLI for everything thereafter (see [bootstrap-design](../05-bootstrap/bootstrap-design.md) Stage 1).

- Consequences:
  - **Positive**: Same runtime as the managed CLIs — Linuxify can `require()` a managed CLI's modules to inspect them, can use the same `npm`/`npx` the user uses, and can debug Node-level issues with the same tools. Mature ecosystem — every problem we have encountered has a well-trodden npm solution. Static typing catches a large class of refactoring bugs at compile time. TypeScript's adoption is broad enough that the contributor pool is large. Cross-platform by default — Linuxify runs on aarch64, armv7l, and x86_64 Android without per-arch compilation. The [ADR-001](adr-001-use-proot-over-chroot.md) choice of proot is compatible: Node runs inside proot without modification.
  - **Negative**: Node startup latency (~30–50 ms cold) is non-trivial; for a `linuxify doctor` that runs in <3 seconds this is fine, but for a `linuxify run cline --help` that is already paying 40–80 ms of proot startup, the additional Node startup is noticeable. We mitigate by keeping the Linuxify core import graph small (lazy-loading subsystems that are not needed for every command) and by offering a "Direct" launcher variant (see [ADR-004](adr-004-shell-launchers-over-symlinks.md)) for performance-critical tools. Node's memory footprint (~30–50 MB RSS for a typical Linuxify command) is acceptable on modern Android devices but would be a problem on a phone with <2 GB RAM. The chicken-and-egg bootstrap adds ~1 second to the very first `linuxify init` and is a small maintenance burden (the shell script must be kept in sync with the Node installer).
  - **Neutral**: The choice is reversible in principle (a Rust rewrite could ship as `linuxify` v2 with a migration path) but the ecosystem alignment makes this unlikely. The Vitest test framework (see [testing-strategy](../12-testing/testing-strategy.md) §3) is the natural choice given this decision.

- Alternatives Considered:
  - **Rust**: rejected because the runtime mismatch with the managed Node CLIs would double the contributor surface, and because per-platform cross-compilation and signing is more operationally complex than npm distribution. Reserved for a possible future `linuxify-launcher` micro-binary if launcher latency ever becomes unacceptable.
  - **Go**: rejected because the runtime mismatch is the same as Rust's, and because Go's ecosystem for Node-tooling introspection (parsing `package.json`, walking `node_modules`) is weaker than Node's own.
  - **Python**: rejected because Python startup is slower than Node, the Python-on-Android story (even via Termux) is more fragmented, and the typing story (mypy/pyright) is less mature than TypeScript's.
  - **Shell (POSIX sh)**: rejected because the complexity of Linuxify (state management, TOML parsing, JSON schema validation, AST patching) is far beyond what maintainable shell can express.
