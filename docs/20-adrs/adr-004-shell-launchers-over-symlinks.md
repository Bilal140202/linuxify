# ADR-004: Shell-script launchers over bare symlinks

- Status: accepted
- Date: 2025-01-12
- Deciders: Linuxify core team
- Context: When a user runs `cline` from a Termux shell, something in `$PATH` must intercept that command, enter the proot Ubuntu environment, set up the correct environment variables (merged from Termux, proot, and the package's declared `env:` block), bind-mount the user's working directory, and `exec` the real `cline` binary. The question is what that "something" is. Two options were seriously considered. A **shell-script launcher** is a small (10–30 line) POSIX shell script placed at `$PREFIX/bin/<name>` that calls `exec linuxify run <name> -- "$@"`; Linuxify's `linuxify run` subsystem then does all the proot entry, env setup, and bind-mounting, and finally execs the real binary. A **bare symlink** is a symlink at `$PREFIX/bin/<name>` pointing directly at the real binary inside the distro (e.g., `~/.linuxify/distros/ubuntu/home/linuxify/.local/.../node_modules/cline/dist/cli.js`), bypassing `linuxify run` entirely.

  The forces favoring the shell-script approach are: (1) **preflight checks** — Linuxify needs to verify the distro is installed, the package is installed, the runtime version is correct, and all patches are still applied, *before* the binary runs; a bare symlink skips all of this and the binary fails with a confusing "library not found" or "wrong platform" error. (2) **environment merging** — the binary needs `PATH`, `NODE_PATH`, `HOME`, `TERM`, and package-specific env vars (e.g., `CLINE_PLATFORM=linux`) set up correctly; a bare symlink runs in the raw Termux environment, which lacks these. (3) **proot entry** — the binary *must* run inside proot to see `/proc`, `/sys`, glibc; a bare symlink does not enter proot, and the binary fails immediately. (4) **working-directory binding** — Linuxify bind-mounts the user's cwd into the proot so the binary sees the same files; a bare symlink does not. (5) **debuggability** — a shell script is `cat`-able (`cat $PREFIX/bin/cline` shows exactly what it does), while a symlink is opaque.

  The forces favoring the bare-symlink approach are: (1) **performance** — a shell script adds ~5 ms of shell startup plus the cost of `linuxify run` (preflight + proot entry, ~40–80 ms total); a bare symlink enters proot only if the binary itself does so, which it does not. For a tool invoked hundreds of times per minute (e.g., a code formatter on save), this overhead is unacceptable. (2) **simplicity** — a symlink is one filesystem operation; a shell script is a file with content that must be generated, versioned, and regenerated when the distro or runtime changes.

- Decision: Linuxify uses **shell-script launchers** as the default and recommended launcher variant. The shell script is generated by the launcher subsystem (see [launcher-architecture](../06-launcher/launcher-architecture.md) §2) at `linuxify add` time and placed at `$PREFIX/bin/<name>`. The script's body is `exec linuxify run <name> -- "$@"` plus a shebang and a comment pointing to the docs. The script is regenerated whenever the distro is switched, the runtime is upgraded, or the package is patched, so it never goes stale. For the performance-sensitive minority of tools, Linuxify offers a **Direct launcher variant** (a bare symlink) that bypasses `linuxify run` entirely; this is opt-in per-package via `launcher: direct` in the package YAML, and the package's `doctor:` block is responsible for verifying that the direct path still works (since it skips Linuxify's preflight).

- Consequences:
  - **Positive**: Every `linuxify`-managed tool invocation goes through `linuxify run`, which means preflight checks run on every invocation (a broken distro is detected at `cline` time, not three minutes into a coding session), env merging is consistent (no "it works when I run `linuxify run cline` but not when I run `cline`" bug class), and proot entry is guaranteed. The shell script is inspectable: a user who suspects launcher misbehavior can `cat $PREFIX/bin/cline` and see exactly what it does, which builds trust and makes bug reports more actionable. Working-directory binding "just works" because `linuxify run` does it; a bare symlink would leave the binary looking at `/data/data/com.termux/files/home` instead of the user's project.
  - **Negative**: ~40–80 ms of per-invocation overhead (mostly proot startup, not the shell script itself — the shell script is ~5 ms). For interactive `cline` use this is invisible; for a formatter invoked on every save it is noticeable. The Direct variant addresses this but trades away the preflight safety net — a Direct-launched tool whose distro has been corrupted will fail with a confusing error rather than a clean `linuxify`-preflight message. The shell script must be regenerated on distro/runtime/patch changes; if regeneration is skipped (e.g., a user manually edits `$PREFIX/bin/cline`), the launcher goes stale and the user gets confusing behavior. We mitigate by making `linuxify doctor` check that every launcher matches the expected hash (see [doctor-engine](../07-doctor/doctor-engine.md) §3.6).
  - **Neutral**: A third variant, **Custom**, allows a user-supplied shell script at `~/.linuxify/launchers/<name>.sh` for exotic cases (non-standard proot invocation, pre/post hooks, cross-distro targeting). This is power-user territory and not the default.

- Alternatives Considered:
  - **Bare symlinks as the default**: rejected because the preflight, env-merge, and proot-entry guarantees are load-bearing for the "it just works" experience that is Linuxify's core value proposition. Relegated to the opt-in Direct variant for the performance-sensitive minority.
  - **Compiled binary launchers** (a tiny C or Rust stub that execs `linuxify run`): rejected because the ~5 ms shell-script overhead does not justify the operational complexity of cross-compiling, signing, and distributing a binary per platform.
  - **No launcher, require users to type `linuxify run cline` explicitly**: rejected because it violates the "verb-like, first-class developer tool" branding in the [project context](../../.agent-context.md) §1 — `cline` should be a first-class command, not a subcommand of `linuxify`.
