# ADR-005: Opt-in telemetry (off by default)

- Status: accepted
- Date: 2025-01-15
- Deciders: Linuxify core team
- Context: Linuxify is a CLI tool that runs on personal Android devices, executes third-party code, applies source patches, and is installed by users who may not be security experts. To improve the product, the maintainers need real-world signal: which commands are used, which packages are installed, which errors are most common, which Android versions cause the most proot segfaults, how long `linuxify init` takes in the field. Without this signal, the maintainers are flying blind — fixing bugs they happen to hit rather than bugs users actually hit. The question is how to collect this signal in a way that respects the user's privacy, complies with relevant regulation, and maintains user trust.

  Three options were considered. **Opt-in telemetry (off by default)** means no data is collected until the user explicitly runs `linuxify config telemetry true` or answers "yes" to a first-run prompt; the default is "no." **Opt-out telemetry (on by default)** means data is collected from every install until the user explicitly runs `linuxify config telemetry false`; the default is "yes." **No telemetry** means no data is ever collected, even anonymously; the maintainers rely entirely on GitHub issues and Discord for signal.

  The forces favoring opt-out are data volume: a 5–10× higher participation rate (industry data suggests ~10% of users opt in when asked, vs. ~80% who never bother to opt out) means dramatically more signal, which means better prioritization. The forces favoring opt-in are trust and regulation: Linuxify runs on personal devices, executes third-party code, and patches that code — a user who discovers telemetry is on by default may reasonably wonder what else Linuxify is doing without asking. GDPR (EU), CCPA (California), LGPD (Brazil), and PIPL (China) all favor opt-in for non-essential data collection, and the regulatory trend is toward stricter consent requirements. The forces favoring no telemetry are simplicity: no privacy policy, no anonymization pipeline, no queue file, no data-deletion infrastructure, no risk of a "Linuxify was caught collecting data it said it wasn't" incident.

- Decision: Linuxify uses **opt-in telemetry, off by default** (encoded as PRD functional requirement [FR-052](../01-product/prd.md)). On first run of any `linuxify` subcommand, the CLI prints a one-time prompt asking the user to opt in; the default is `N` (off), and the user must explicitly type `y` to enable collection. There is no nag screen, no periodic re-prompt, no "are you sure?" — the user's choice is final until they explicitly change it via `linuxify config telemetry <bool>`. The full telemetry design — what is collected, what is never collected, the anonymization pipeline, the local queue at `~/.linuxify/telemetry/queue.jsonl`, the data-deletion commands — is documented in [telemetry-privacy](../24-telemetry/telemetry-privacy.md). The opt-in decision is referenced from the [security model](../13-security/security-model.md) §1 as a load-bearing trust commitment.

- Consequences:
  - **Positive**: User trust is preserved — a user who reads the docs and sees "telemetry is opt-in, off by default" is more likely to install Linuxify than one who sees "telemetry is on by default but you can turn it off." GDPR compliance is straightforward: the lawful basis is **consent** (opt-in), which is the strictest basis and the easiest to satisfy; the right to deletion is fulfilled by `linuxify telemetry purge-remote`; the right to access is fulfilled by `linuxify telemetry export`. CCPA compliance is trivial: there is nothing to opt out of unless the user has opted in. The "Linuxify was caught collecting data it said it wasn't" risk is structurally eliminated: the code path that emits telemetry is gated on a single config flag, and that flag defaults to `false`.
  - **Negative**: Data volume is lower than opt-out would provide. We expect ~10–15% opt-in based on comparable open-source CLI tools (Rust's rustup, Homebrew's analytics-on-by-default-but-anonymous). This is enough for gross signal (which packages are popular, which errors are common) but marginal for fine-grained signal (e.g., "what percentage of armv7l users hit this specific proot segfault"). We accept this trade-off because the alternative (opt-out) was deemed more costly in trust than its data was worth. Some users will be confused by the first-run prompt and will not understand what they are consenting to; we mitigate with a clear prompt that names exactly what is collected and links to the [privacy policy](../24-telemetry/telemetry-privacy.md).
  - **Neutral**: The `TELEMETRY_DISABLED` exit code (28, see [cli-specification](../03-cli/cli-specification.md) §6) is emitted to stderr when a user runs `linuxify telemetry export` or similar without telemetry enabled — informational, not an error. The opt-in decision is reversible: if a future version of Linuxify switches to opt-out (with anonymization safeguards), this ADR would be superseded by a new one. No such switch is currently planned.

- Alternatives Considered:
  - **Opt-out telemetry (on by default)**: rejected because the trust cost on a personal-device tool that patches third-party code is too high. The data-volume benefit, while real, does not justify the reputational risk of a "Linuxify spies on you by default" perception.
  - **No telemetry**: rejected because the maintainers need real-world signal to prioritize fixes, and GitHub issues + Discord are too biased toward the most vocal users to be representative. The opt-in design captures the signal of users who care enough to opt in, which is a useful (if self-selected) sample.
  - **Telemetry on by default for anonymous aggregate counts only, opt-in for per-user data**: rejected because the distinction is too subtle to communicate clearly in a first-run prompt, and because "anonymous aggregate counts" can be re-identified in edge cases (a single armv7l user on Android 9 in a specific region is identifiable). The cleaner line is "off by default, opt in for everything."
