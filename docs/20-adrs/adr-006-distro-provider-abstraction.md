# ADR-006: DistroProvider abstraction (pluggable distros)

- Status: accepted
- Date: 2025-01-18
- Deciders: Linuxify core team
- Context: [ADR-001](adr-001-use-proot-over-chroot.md) settled that Linuxify would use `proot-distro` as its default mechanism for entering a Linux userland on Android. ADR-001 also noted, almost in passing, that "the DistroProvider interface is pluggable, so a future chroot-based or QEMU-based provider can be added without re-architecting Linuxify." This ADR formalizes that interface and explains why we are paying the cost of an abstraction *now*, when v1 ships exactly one provider (Ubuntu via proot-distro), rather than waiting until a second provider is actually needed. The question is whether to (a) hardcode Ubuntu + proot-distro in the bootstrap and run subsystems and refactor later if/when a second distro is requested, or (b) define a `DistroProvider` interface from day one and route every distro operation through it.

  The forces favoring the hardcoded approach are simplicity and speed: a `prootDistroInstall("ubuntu")` call is two lines, the abstraction is zero, and there is no risk of designing an interface that turns out to be the wrong shape once a second distro (say, Alpine) is added. The forces favoring the abstraction are twofold. First, the surface area that depends on "which distro is installed" is large — bootstrap Stage 3 downloads and unpacks the rootfs, the launcher subsystem enters the distro on every invocation, the doctor subsystem checks distro health, the patcher subsystem applies patches *inside* the distro, the package installer runs `apt install` inside the distro — and hardcoding `ubuntu` in each of these five subsystems creates five distinct sites that must be touched in lockstep when a second distro arrives. Second, the Termux community already has working proot-distro recipes for Debian, Arch, and Alpine, and the project context (§9) lists these as v1-supported distros; the second distro is not hypothetical, it is a published v1 goal, so the abstraction is not speculative but rather a planned-for necessity.

- Decision: Linuxify defines a `DistroProvider` interface in `src/distros/provider.ts` (see [AI Build Guide](../00-executive/ai-build-guide.md) §7 for the directory tree) that exposes the following operations: `name()`, `availableVersions()`, `install(version, opts)`, `uninstall()`, `isInstalled()`, `login(argv, opts)`, `exec(argv, opts)`, `packageManager()`, `healthCheck()`. Every subsystem that touches a distro — bootstrap, launcher, doctor, patcher, package installer — depends on `DistroProvider`, never on `proot-distro` directly. v1 ships one provider, `UbuntuProvider`, implemented on top of `proot-distro login ubuntu`. The `linuxify use <distro>` command (see [CLI spec](../03-cli/cli-specification.md) §3) switches the active provider by writing to `~/.linuxify/state.json`; the next `linuxify run` invocation reads the active provider name from state and instantiates the matching class from a registry. Adding Debian in v1.1 means writing `DebianProvider` (≈120 lines, mostly inherited from `UbuntuProvider` since both use `apt`) and registering it; no subsystem outside `src/distros/` changes.

- Consequences:
  - **Positive**: The interface forces every subsystem to think in terms of "the active distro" rather than "Ubuntu," which keeps the codebase honest about what is distro-specific (package names, filesystem layout, default shell) and what is generic (proot entry, bind-mounting, env merging). Adding a distro is a bounded change — a single new file in `src/distros/` plus a registry entry — that cannot accidentally leak `ubuntu` strings into the launcher or doctor. A future chroot provider (for rooted users who want native performance) or QEMU provider (for x86-only tools on ARM) slots in without disturbing existing code. The interface makes testing dramatically easier: a `MockDistroProvider` lets unit tests run on CI without invoking proot at all, which is essential because proot does not work on GitHub Actions Linux runners (they lack ptrace permission).
  - **Negative**: The interface adds one indirection per distro operation. For a hot path like `linuxify run`, this is one extra function call (negligible). For maintainability, the interface is a contract that must be respected — a contributor who bypasses it and calls `proot-distro` directly from the launcher creates a hidden Ubuntu dependency that breaks when the user switches to Alpine. We mitigate with a lint rule (`no-restricted-syntax` banning the string `proot-distro` outside `src/distros/`) and with a doctor check that scans the codebase at release time. There is also the risk that the interface is the wrong shape: if v2 needs, say, a `snapshot()` operation for fast rollback, the interface must be extended and every provider updated. We accept this risk because adding a method is backward-compatible (existing callers ignore the new method).
  - **Neutral**: The interface is TypeScript-first (uses `async` and `Promise<Result<T, E>>`); a future non-TypeScript provider (e.g., a Rust-compiled binary provider for performance) would need a JSON-over-stdio adapter. This is documented as a v3+ possibility in [vision-extension](../19-future/vision-extension.md).

- Alternatives Considered:
  - **Hardcode Ubuntu, refactor later**: rejected because the "later" refactor would touch five subsystems simultaneously, with high risk of regressions, and because v1 already promises multiple distros so the refactor is not optional but merely deferred.
  - **Define the interface but ship only Ubuntu, no pluggability in the CLI**: rejected because `linuxify use <distro>` is a published v1 command (see [CLI spec](../03-cli/cli-specification.md) §3) and the command requires the registry to exist.
  - **Use a runtime plugin system (load providers from `~/.linuxify/providers/`)**: rejected for v1 as over-engineered; providers are compiled-in for now, and a runtime plugin system can be layered on later without changing the interface.
