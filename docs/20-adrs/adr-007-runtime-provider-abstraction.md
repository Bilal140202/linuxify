# ADR-007: RuntimeProvider abstraction (pluggable runtimes)

- Status: accepted
- Date: 2025-01-18
- Deciders: Linuxify core team
- Context: Linuxify's package definitions (see [ADR-002](adr-002-yaml-package-definitions.md), [package-spec](../09-registry/package-spec.md)) declare a `runtime:` field — `node`, `python`, `rust`, `go`, `bun`, or `deno` — that tells Linuxify which language runtime a managed CLI requires. Linuxify must install, version, upgrade, and verify that runtime inside the proot distro before the CLI itself can be installed. The question is whether to (a) hardcode Node-only support in v1 (since the most popular managed CLIs — Cline, Codex, Aider's JS tooling — are Node applications) and add Python/Rust/Go later as ad-hoc branches, or (b) define a `RuntimeProvider` interface from day one and route every runtime operation through it. This ADR is the runtime-side companion to [ADR-006](adr-006-distro-provider-abstraction.md); the two interfaces are deliberately symmetric in shape so that a contributor who understands one understands the other.

  The forces favoring the Node-only approach are familiar: less code, no speculative abstraction, faster time-to-v1. The forces favoring the abstraction are stronger here than in the distro case for two reasons. First, the *variety* of runtimes is greater than the variety of distros: v1 promises Node, Python, and Git (see [project context](../../.agent-context.md) §9), and v1.1+ adds Rust, Go, Bun, and Deno. Each runtime has a different version manager (`nvm` for Node, `pyenv` for Python, `rustup` for Rust, `gvm` for Go), a different install procedure, a different mechanism for "set the active version," and a different way to install global packages (`npm install -g`, `pip install`, `cargo install`, `go install`). Hardcoding Node in five subsystems means that adding Python touches all five plus introduces a parallel `python-only` code path that does not share infrastructure. Second, runtimes are *per-package*: a single Linuxify installation may have Cline (Node 20), Aider (Python 3.12), and a future Rust tool all installed simultaneously, each with its own runtime version pinned. This means the runtime selection is not a global setting (like the active distro) but a per-package lookup, which makes the abstraction load-bearing rather than optional.

- Decision: Linuxify defines a `RuntimeProvider` interface in `src/runtimes/provider.ts` with operations: `name()`, `availableVersions()`, `install(version, opts)`, `uninstall(version)`, `isActive(version)`, `exec(version, argv, opts)`, `installGlobalPackage(version, pkg, opts)`, `uninstallGlobalPackage(version, pkg)`, `healthCheck()`. Each managed runtime has a class implementing this interface: `NodeRuntimeProvider` (built on a vendored `nvm`-like installer), `PythonRuntimeProvider` (`pyenv`-style), `RustRuntimeProvider` (`rustup`-style), `GoRuntimeProvider`, `BunRuntimeProvider`, `DenoRuntimeProvider`. The package installer consults the package's `runtime:` field, looks up the matching provider in the runtime registry, calls `install(runtime_min_version)` if the version is not already present, then calls `installGlobalPackage(version, package)` to install the CLI itself. The `linuxify doctor` command iterates over all installed packages and asks each package's runtime provider to run `healthCheck()`.

- Consequences:
  - **Positive**: Adding a new runtime is a bounded change — one new file in `src/runtimes/` plus a registry entry — that cannot leak `npm` or `node` strings into code paths that should be runtime-agnostic. The per-package runtime lookup is explicit: a contributor reading the package installer sees `provider.installGlobalPackage(version, pkg)` rather than a switch statement on `runtime`, which makes the code obviously correct. Testing is dramatically easier: a `MockRuntimeProvider` lets unit tests verify the installer's logic without actually installing Node or Python. The interface also enables a future "runtime shim" mode where Linuxify intercepts `node`/`python` invocations inside proot to apply per-package version pinning, without changing any caller.
  - **Negative**: The interface is more complex than `DistroProvider` because runtimes are versioned and per-package rather than singleton. The `installGlobalPackage` operation in particular must handle three different package-spec shapes (npm's `pkg@version`, pip's `pkg==version`, cargo's `pkg` plus `--version`) and the interface abstracts over all three, which means a contributor adding a new runtime must understand the abstraction. We mitigate by documenting the expected behavior in the interface's JSDoc and by providing a `BaseRuntimeProvider` abstract class that handles 80% of the boilerplate. There is also a performance cost: every runtime operation is one extra function call, but this is negligible compared to the cost of the runtime operation itself (an `npm install -g` is hundreds of milliseconds; the indirection is microseconds).
  - **Neutral**: The interface uses `Promise<Result<T, E>>` for error handling, matching `DistroProvider`. A future non-TypeScript runtime (e.g., a JVM runtime for Java-based CLIs) would need a JSON-over-stdio adapter; this is documented as a v2+ possibility.

- Alternatives Considered:
  - **Hardcode Node, branch for Python later**: rejected because per-package runtime selection means the branching happens in five places (bootstrap, package installer, doctor, launcher, patcher) and produces a parallel-maintenance burden.
  - **Use the host system's runtime managers directly (apt install nodejs)**: rejected because apt's Node version is always stale on Ubuntu LTS, and because per-package version pinning is impossible without a version manager.
  - **Single runtime per Linuxify install (no per-package pinning)**: rejected because Cline may require Node 20 while a future package requires Node 22, and forcing the user to upgrade everything at once defeats the "it just works" promise.
