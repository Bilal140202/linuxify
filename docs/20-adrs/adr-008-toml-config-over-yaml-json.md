# ADR-008: TOML for `config.toml` (user config) over YAML or JSON

- Status: accepted
- Date: 2025-01-19
- Deciders: Linuxify core team
- Context: Linuxify has two distinct configuration surfaces. The first is **package definitions** (`cline.yml`, `codex.yml`), which are contributed by the open-source community in large numbers, are nested and array-heavy (multiple `patches:`, `doctor:` blocks per file), and were settled as YAML by [ADR-002](adr-002-yaml-package-definitions.md). The second is the **user configuration file** at `~/.linuxify/config.toml`, which is hand-edited by an individual user on their Android device, is flat or shallowly-nested (a handful of top-level keys like `telemetry`, `default_distro`, `parallel_jobs`, `log_level`, each with a few scalar values), and is read on every `linuxify` invocation. This ADR concerns only the second surface. The question is whether to reuse YAML (consistency with package definitions, one parser in the dependency tree), use TOML (better ergonomics for flat config, comments, less surprising type coercion), or use JSON (universality, native to Node).

  The forces favoring YAML are ecosystem consistency — a contributor who has read `cline.yml` already knows the syntax — and the avoidance of a second parser in the dependency tree. The forces favoring TOML are threefold. First, **whitespace sensitivity is a poor fit for hand-edited flat config on a phone**: a user editing `~/.linuxify/config.toml` in Termux's `nano` or `vim` is far more likely to mis-indent a YAML block than to mis-bracket a TOML section, and the resulting error messages from a TOML parser are typically clearer ("line 12: expected `=` after key") than from a YAML parser ("could not find expected ':' at line 12"). Second, **TOML's type coercion is less surprising**: YAML parses `no`, `yes`, `on`, `off` as booleans (the "Norway problem"), which is a footgun for a config file where a user might write `telemetry = no` and get the boolean `false` (correct by luck) but might also write `version = 1.0` and get the string `"1.0"` rather than the float (depending on YAML parser version). TOML has no such ambiguity: `telemetry = false` is a boolean, `version = "1.0"` is a string, `port = 8080` is an integer, full stop. Third, **TOML has a single canonical parser** (`@iarna/toml`, used by `npm` itself) with predictable behavior, while YAML has multiple Node parsers (`js-yaml`, `yaml`) with subtly different behavior on edge cases.

  The forces favoring JSON are universality (every language has a JSON parser) and native Node support (`require('config.json')` works without a dependency). The forces against JSON are no comments, strict quoting, no trailing commas, and a hostility to hand-authoring that ADR-002 already rejected for package definitions.

- Decision: Linuxify uses **TOML 1.0** for `~/.linuxify/config.toml`, parsed and serialized via `@iarna/toml` (the same library `npm` uses for `.npmrc`-adjacent files). The schema is defined in Zod (see [ADR-015](adr-015-zod-for-schema-validation.md)) in `src/config/schema.ts` and validated on every load. The file is read once per `linuxify` invocation and cached in-memory; mutation happens via `linuxify config <key> <value>` (which rewrites the file atomically — write to temp, rename). The CLI never writes YAML to `~/.linuxify/`; YAML is reserved for package definitions, where its array-of-objects ergonomics win (per ADR-002). Configuration overrides apply in the order: built-in defaults (`src/config/defaults.ts`) → `~/.linuxify/config.toml` → environment variables (`LINUXIFY_*`) → command-line flags, with later layers winning. See [CLI spec](../03-cli/cli-specification.md) §5 for the override precedence table.

- Consequences:
  - **Positive**: Hand-editing `~/.linuxify/config.toml` on a phone is forgiving — section headers (`[telemetry]`), `key = value` lines, and `# comments` are all natural to type in a mobile text editor. The lack of surprising type coercion means a user who writes `parallel_jobs = 4` gets the integer `4`, not a string. The `@iarna/toml` parser is mature, fast, and produces line-precise error messages, which makes `linuxify config` validation errors immediately actionable. TOML's comment support (`#`) lets users annotate their own config ("I disabled telemetry because…"), which is impossible in JSON. The choice aligns with industry peers: `cargo` uses TOML for `Cargo.toml`, `pip` uses TOML for `pyproject.toml`, `npm` uses INI-ish TOML-adjacent for `.npmrc` — developers already know the format.
  - **Negative**: Two parsers in the dependency tree (`@iarna/toml` for config, `js-yaml` for package definitions) is a small cost — both are <100 KB minified, both are well-maintained, and the alternative (forcing package definitions into TOML) was rejected by ADR-002. A contributor who works on both surfaces must context-switch between the two syntaxes; we mitigate by keeping the two files visually distinct (YAML's `key:` vs TOML's `key =`) and by providing a `linuxify config edit` command that opens the file in the user's `$EDITOR` with a syntax-comment banner. TOML's handling of arrays-of-objects (`[[foo]]`) is more verbose than YAML's nested blocks, which is why we do not use TOML for package definitions — but for the flat config surface, TOML's array syntax (`disabled_packages = ["cline", "aider"]`) is fine.
  - **Neutral**: The format choice is reversible (a v2 could read both TOML and YAML for config), but no contributor has requested this. The Zod schema is the source of truth; the TOML file is just the serialization.

- Alternatives Considered:
  - **YAML for config too (consistency)**: rejected because hand-editing flat YAML on a phone is more error-prone than hand-editing TOML, and because the "Norway problem" and similar type-coercion surprises are unacceptable in a config file that controls security-sensitive settings like `telemetry`.
  - **JSON for config**: rejected for the same reasons ADR-002 rejected it for package definitions — no comments, strict quoting, hostile to hand-authoring.
  - **JSON5 / JSONC**: rejected because they are less widely known than TOML and offer no compelling advantage for the flat config shape.
  - **A custom key=value format (`.linuxifyrc`)**: rejected because it cannot represent nested config (e.g., `[telemetry] endpoint = ...`) without inventing section syntax, at which point we have reinvented TOML badly.
