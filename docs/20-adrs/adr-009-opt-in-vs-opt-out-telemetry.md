# ADR-009: Opt-in vs. opt-out telemetry — deeper dive (companion to ADR-005)

- Status: accepted
- Date: 2025-01-20
- Deciders: Linuxify core team
- Context: [ADR-005](adr-005-opt-in-telemetry.md) decided that Linuxify's telemetry would be **opt-in, off by default**. This ADR is a companion document that does not revisit the decision but records the deeper analysis of the opt-in-vs-opt-out trade-off that informed it, with particular attention to the GDPR, CCPA, LGPD, and PIPL regulatory landscape. ADR-005 was deliberately short (the Nygard template favors brevity); this ADR exists because the regulatory analysis was substantive enough that future maintainers considering a switch to opt-out need to see the full reasoning rather than a one-line "GDPR favors opt-in."

  The fundamental trade-off is **data volume vs. trust and consent-law compliance**. Industry data from comparable open-source CLI tools (rustup, Homebrew, Deno, Bun) suggests opt-out telemetry captures 5–10× more events than opt-in, because ~80–90% of users never bother to opt out of a default-on setting, while only ~10–15% of users bother to opt in to a default-off setting. For a project that needs to prioritize bug fixes, this 5–10× multiplier is the difference between "we know which Android versions segfault proot" and "we are guessing." The counter-force is that Linuxify is not a generic CLI tool — it runs on personal Android devices, executes third-party code, and applies source patches to that code. A user who discovers that Linuxify has been phoning home by default may reasonably wonder what *else* Linuxify is doing without asking, and that trust damage is hard to repair. The Homebrew analytics incident of 2018 (where users discovered analytics was on by default and a vocal subset revolted) is the canonical cautionary tale; we prefer to avoid repeating it.

  The regulatory landscape reinforces the trust argument. **GDPR (EU)** requires a lawful basis for processing personal data; for non-essential telemetry (which Linuxify's is — the tool works fine with telemetry off), the only lawful basis that does not require an elaborate legitimate-interest assessment is **consent**, and GDPR-grade consent must be "freely given, specific, informed, and unambiguous" — which in practice means opt-in, not opt-out. **CCPA (California)** is less strict (it allows opt-out for "sale" of data, and Linuxify does not sell data), but CCPA's definition of "sale" is broad enough that sharing telemetry with a third-party analytics provider could trigger opt-out requirements; opt-in sidesteps this entirely. **LGPD (Brazil)** mirrors GDPR's consent requirement. **PIPL (China)** requires separate consent for cross-border data transfer, which opt-in handles cleanly. The regulatory trend (GDPR's 2023 guidance on "dark patterns," the EU AI Act's transparency requirements, the FTC's 2024 enforcement actions against default-on telemetry) is unambiguously toward stricter consent, not looser.

- Decision: This ADR reaffirms ADR-005 without modification: Linuxify uses **opt-in telemetry, off by default**, gated on a single config flag (`telemetry = false` in `~/.linuxify/config.toml` by default, per [ADR-008](adr-008-toml-config-over-yaml-json.md)). The first-run prompt is the only nag — there is no periodic re-prompt, no "are you sure?", no degraded experience for users who decline. The regulatory compliance strategy is documented in [telemetry-privacy](../24-telemetry/telemetry-privacy.md) §4: the lawful basis is consent (opt-in), the right to access is fulfilled by `linuxify telemetry export`, the right to deletion by `linuxify telemetry purge-remote`, the right to object by `linuxify config telemetry false`. Anonymization (hashing user IDs with a rotating salt, truncating IP addresses, redacting file paths) is applied at *emit* time, before the event leaves the device, so even if the ingest server is compromised, no personal data is recoverable.

- Consequences:
  - **Positive**: GDPR compliance is structurally guaranteed — there is no data to delete or export until the user opts in, at which point the export and purge commands cover the rights. CCPA compliance is trivial. User trust is preserved: the docs say "off by default" and the code matches the docs. The "Linuxify was caught collecting data it said it wasn't" risk is structurally eliminated because the code path that emits telemetry is gated on a single config flag, and that flag is `false` until the user explicitly sets it `true`. Auditors (and curious users) can verify this by reading `src/telemetry/client.ts`, which is ~150 lines and has a single `if (!config.telemetry) return;` guard at the top of every emit function.
  - **Negative**: Data volume is lower than opt-out would provide. We expect ~10–15% opt-in, which is enough for gross signal (which packages are popular, which errors are common) but marginal for fine-grained signal (e.g., "what percentage of armv7l users hit this specific proot segfault on Android 14"). We mitigate by making the opt-in prompt informative (it names exactly what is collected and links to the privacy policy) rather than coercive, and by ensuring that users who *do* opt in are not penalized (no degraded performance, no nag screens, no "thanks for opting in!" banners). Some users will be confused by the prompt and choose `N` out of caution; we accept this as the cost of an honest default.
  - **Neutral**: The decision is reversible — a future ADR could switch to opt-out (with anonymization safeguards) if the maintainers decide the data-volume benefit outweighs the trust cost. No such switch is currently planned, and any such switch would require a new ADR that explicitly supersedes ADR-005 and this one.

- Alternatives Considered:
  - **Opt-out telemetry (on by default)**: rejected (per ADR-005) because the trust cost on a personal-device tool that patches third-party code is too high, and because GDPR-grade consent is harder to argue for opt-out than opt-in.
  - **No telemetry**: rejected (per ADR-005) because the maintainers need real-world signal to prioritize fixes.
  - **Opt-out for anonymous aggregate counts, opt-in for per-user data**: rejected (per ADR-005) because the distinction is too subtle to communicate in a first-run prompt, and because "anonymous aggregate counts" can be re-identified in edge cases.
  - **Opt-in but with a periodic re-prompt (every N major versions)**: rejected because re-prompting erodes trust ("I already said no, why are you asking again?") and because users who opted in at v0.1 may forget by v0.5 and be surprised to find telemetry still on; the clean rule is "your choice is final until you explicitly change it."
