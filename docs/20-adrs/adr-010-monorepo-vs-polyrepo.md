# ADR-010: Monorepo vs. polyrepo (start monorepo, split on friction)

- Status: accepted
- Date: 2025-01-21
- Deciders: Linuxify core team
- Context: Linuxify at v1 consists of several distinct artifacts that *could* live in separate GitHub repositories: the TypeScript CLI source (`linuxify`), the documentation set (this `linuxify-docs` repo), the v1 package registry (a git repo of YAML files, per [ADR-011](adr-011-git-based-registry-v1.md)), the example plugins, the build scripts, and (eventually) the sync server and registry server components (per [ADR-014](adr-014-bsl-for-server-side-components.md)). The question is whether to start as a **monorepo** (everything in one repo, `linuxify/linuxify`) or as a **polyrepo** (separate repos from day one: `linuxify/cli`, `linuxify/docs`, `linuxify/registry`, `linuxify/plugins-example`, etc.).

  The forces favoring a monorepo are well-known and well-documented (the "Why Monorepos" essays from Google, Facebook, and Babel). First, **atomic cross-cutting changes**: a change to the CLI's command surface requires updates to the CLI source, the CLI spec, the command reference, the AI Build Guide, and possibly a package-definition example; in a monorepo this is one PR, in a polyrepo it is four coordinated PRs that must be merged in order. Second, **shared tooling**: the ESLint config, Prettier config, Vitest config, tsconfig, and CI workflows are identical across the CLI and the example plugins; a monorepo lets these live in one place, while a polyrepo requires either a shared config package (which then needs its own versioning and release) or duplicated config that drifts. Third, **discoverability**: a new contributor who clones `linuxify/linuxify` sees the entire project — CLI, docs, registry, plugins — in one tree, which makes it easy to understand the relationships; a polyrepo requires the contributor to discover and clone five repos. Fourth, **issue and PR tracking**: a single GitHub issue tracker means a bug report that touches both CLI and docs is one issue, not two cross-referenced issues.

  The forces favoring a polyrepo are: cleaner access control (the docs repo can be open to community edits while the CLI repo requires maintainer review for code), smaller clone size (a docs-only contributor does not need to download the CLI source), independent release cadences (the registry can update hourly while the CLI releases monthly), and license flexibility (the CLI can be MIT while the sync server is BSL, per [ADR-014](adr-014-bsl-for-server-side-components.md), without license-file confusion).

- Decision: Linuxify starts as a **monorepo** at `linuxify/linuxify` containing the CLI source, the docs, the v1 registry, the example plugins, and the build scripts. The sync server and registry server (when they land in v2) will also live in the monorepo under `server/` subdirectories, with their own `package.json` and their own BSL `LICENSE` file (per ADR-014), so the license heterogeneity is handled at the directory level rather than the repo level. We will split a sub-project into its own repo **only when friction emerges** — concrete triggers being: (a) a sub-project needs a different release cadence that the monorepo's release workflow cannot accommodate (e.g., the registry needs hourly updates and the monorepo release process takes a day); (b) a sub-project's clone size becomes a contributor obstacle (e.g., the docs hit 500 MB because of rendered PDFs); (c) access control diverges (e.g., the registry becomes community-editable in a way that the CLI's maintainer-review model cannot support). The docs are an exception: they currently live in `linuxify/linuxify-docs` as a separate repo *only because* this documentation effort was bootstrapped before the CLI repo existed; the docs will be merged into `linuxify/linuxify/docs/` at the v0.1.0 release.

- Consequences:
  - **Positive**: Atomic cross-cutting changes (a CLI change that updates the spec, docs, and an example plugin) ship as one PR with one review. Shared tooling (ESLint, Prettier, Vitest, tsconfig) lives in the repo root and is consumed by all sub-projects via relative paths, so it never drifts. A new contributor clones one repo and sees everything. Issue tracking is unified. The "did the docs get updated for this CLI change?" question is answered by the PR template, which has a "docs updated?" checkbox. The monorepo also makes the AI Build Guide (see [ai-build-guide](../00-executive/ai-build-guide.md)) more useful, because an AI agent cloning the repo gets the entire context — CLI, docs, registry, examples — in one operation.
  - **Negative**: Clone size grows over time; we mitigate with shallow clones (`git clone --depth 1`) in CI and with `git sparse-checkout` for contributors who only need a subdirectory. Release complexity is higher: a single `linuxify` release must publish the npm package, update the registry, regenerate the docs site, and tag the repo — but this is true in a polyrepo too (each repo releases independently, but the *coordination* is the same). License heterogeneity (MIT for CLI/docs/plugins/registry, BSL for servers) requires careful `LICENSE` file placement and a `README.md` note in each subdirectory explaining its license; we mitigate with a `scripts/check-licenses.ts` that verifies every subdirectory has the expected `LICENSE` file at release time. CI runtime grows because a change to docs re-runs the CLI's test suite; we mitigate with path-filtered workflows (`.github/workflows/cli.yml` only runs on `src/**` changes).
  - **Neutral**: The "split on friction" policy means we do not commit to monorepo forever. If the registry outgrows the monorepo (e.g., it becomes a large binary store for signed packages), we will split it then. The split is a one-time `git filter-repo` operation plus a redirect, not a rewrite.

- Alternatives Considered:
  - **Polyrepo from day one**: rejected because the coordination cost of four coordinated PRs for every cross-cutting change is too high for a small maintainer team, and because shared tooling drift across repos is a real and recurring problem in every polyrepo project we have observed.
  - **Monorepo forever, never split**: rejected as too rigid; we explicitly reserve the right to split when friction emerges, and we name the triggers so the decision is not ad-hoc.
  - **Monorepo with git submodules for docs and registry**: rejected because submodules add operational complexity (detached HEAD state, sync drift) for no benefit over a plain monorepo.
