# ADR-011: Git-based registry for v1 (over HTTP API from day 1)

- Status: accepted
- Date: 2025-01-22
- Deciders: Linuxify core team
- Context: Linuxify needs a **package registry** — the source of truth for "which packages exist, what versions are available, where to download them, what their signatures are." When a user runs `linuxify add cline`, the CLI must query this registry to resolve the `cline` package, fetch its definition (per [ADR-002](adr-002-yaml-package-definitions.md)), and install it. The question is what the registry *is* in v1. Two options were seriously considered. A **git-based registry** is a plain git repository (e.g., `linuxify/registry`) whose tree contains one YAML file per package (`c/cline.yml`, `c/codex.yml`, `a/aider.yml`) plus an `index.toml` listing all packages and their latest versions; the CLI clones or fetches this repo to `~/.linuxify/registry/` and reads files locally. An **HTTP API registry** is a server (e.g., `registry.linuxify.dev`) that exposes `GET /packages/<name>`, `GET /packages/<name>/versions`, `POST /search?q=...` endpoints; the CLI makes HTTP requests at runtime.

  The forces favoring a git-based registry are simplicity and transparency. A git repo requires zero infrastructure — GitHub hosts it for free, signed commits provide integrity (per [ADR-013](adr-013-ed25519-for-package-signing-v2.md), v2 will add Ed25519 signatures on top), and the entire registry is cloneable by any user for offline inspection. The "API" is `git fetch` + `git log`, which every developer already knows. Updates to the registry (a contributor adding a new package) are PRs against the repo, reviewed by maintainers — the same workflow as code contributions, with the same CI linting. The forces favoring an HTTP API are performance (no clone needed for a single-package lookup) and search (a server can do full-text search across all package definitions, while a git client must read every file). The forces favoring git are stronger in v1 because (a) the package count is small (<100 packages expected in the first year), so a full clone is <2 MB and takes <1 second on a phone, (b) the maintainer team is small and cannot operate a 24/7 HTTP service, and (c) the v1 user base is small and an HTTP service would be over-provisioned.

- Decision: Linuxify v1 uses a **git-based registry** at `linuxify/registry`. The registry repo contains one YAML file per package, organized by first letter (`c/cline.yml`, `c/codex.yml`), plus a top-level `index.toml` mapping package names to their current version and file path. The CLI's `src/registry/git-registry.ts` (see [AI Build Guide](../00-executive/ai-build-guide.md) §7) clones the registry to `~/.linuxify/registry/` on first use and `git fetch`es on every `linuxify add`, `linuxify search`, `linuxify info`, and `linuxify update`. Package *install* (downloading the actual CLI binary or source) still goes through the runtime's native package manager (`npm install -g cline`), so the registry is metadata-only — it tells Linuxify *how* to install, not the bytes themselves. The registry accepts contributions via PR; CI runs `linuxify package lint` on every PR (per [ADR-002](adr-002-yaml-package-definitions.md)) and rejects malformed definitions. The v2 HTTP API registry is documented as future work in [package-registry-future](../19-future/package-registry-future.md) and will be backed by the same git repo (the server reads from a checkout, so there is one source of truth).

- Consequences:
  - **Positive**: Zero infrastructure cost — GitHub hosts the registry for free, with CDN-backed cloning. Full transparency — any user can `git clone https://github.com/linuxify/registry` and inspect every package definition, every change, every maintainer review comment. Offline operation — once the registry is cloned, `linuxify info cline` works without network. Signed history — every commit to the registry is GPG- or SSH-signed by a maintainer, and the CLI verifies signatures against a pinned set of maintainer keys (shipped with the CLI), so a compromised GitHub account cannot inject a malicious package without also compromising a maintainer's signing key. Low operational burden — there is no server to monitor, no database to back up, no incident response for downtime; the registry is "up" whenever GitHub is up.
  - **Negative**: Search is limited — `linuxify search <query>` must read every YAML file in the registry (which is fast for <100 packages, ~10 ms, but would be slow at 10,000 packages). We mitigate with a generated `index.toml` that the CLI reads first, so common queries (by name, by tag) are O(1) lookups; full-text search remains O(N) but is rare. Updates require a `git fetch`, which is fast (<1 second for the v1 registry size) but adds latency to the first `linuxify add` after a registry update; we mitigate with a background fetch triggered by `linuxify doctor`. The registry cannot serve binary packages (e.g., a pre-built rootfs tarball) without LFS, and LFS has bandwidth limits on free GitHub accounts; we avoid this by having the registry be metadata-only and downloading binaries from their original sources (Ubuntu rootfs from cdimage.ubuntu.com, Node from nodejs.org, etc.).
  - **Neutral**: The v2 HTTP API registry will be backed by the same git repo (the server is a thin HTTP layer over a checkout), so the migration is additive — `linuxify add` continues to work via git, while a new `linuxify add --http` flag uses the API for users who want it. The git registry is not deprecated in v2; it remains the canonical source and the offline fallback.

- Alternatives Considered:
  - **HTTP API from day 1**: rejected because the maintainer team cannot operate a 24/7 service, because the v1 package count does not justify a server, and because a git registry is strictly more transparent.
  - **Federated registry (multiple mirrors, like Debian apt)**: rejected for v1 as over-engineered; one GitHub repo is sufficient, and mirrors can be added later by users who clone and republish.
  - **IPFS / decentralized registry**: rejected for v1 because the user experience (waiting for IPFS to find a peer) is worse than `git clone`, and because the v1 user base is too small for a decentralized network to have many seeds.
  - **npm registry (reuse npm as the package source)**: rejected because Linuxify manages non-Node CLIs too (Python, Rust, Go), and because the Linuxify registry carries metadata (patches, env, doctor checks) that npm's package.json does not.
