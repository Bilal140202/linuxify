# ADR-012: DCO (Developer Certificate of Origin) only, no CLA

- Status: accepted
- Date: 2025-01-23
- Deciders: Linuxify core team
- Context: Linuxify accepts external contributions (package definitions, CLI code, docs, plugins) from the open-source community. Every open-source project that accepts contributions must decide how to handle the *intellectual property* of those contributions: who owns the contributed code, what rights the project has to use it, and what rights the contributor retains. Two mechanisms are common. A **CLA (Contributor License Agreement)** is a legal agreement, signed by the contributor (often electronically via cla-assistant.io or similar), that grants the project a license to use the contribution; many CLAs also grant the project a copyright assignment or a patent license. A **DCO (Developer Certificate of Origin)** is a lighter-weight attestation, made by adding a `Signed-off-by: Name <email>` line to each commit, in which the contributor certifies that they have the right to contribute the code under the project's license; the DCO is the same text used by the Linux kernel and is recorded in `Documentation/process/submitting-patches.rst`.

  The forces favoring a CLA are: (a) some corporations require a CLA before their employees can contribute (because the CLA's patent grant protects the company from later patent claims by the contributor), and (b) a CLA with copyright assignment lets the project relicense the code in the future without tracking down every contributor for permission. The forces favoring a DCO are: (a) it is dramatically lower friction — `git commit -s` adds the line, no separate signing step, no account on a third-party service — which means more casual contributors follow through; (b) it requires no legal review to set up (a CLA must be drafted by a lawyer, and a poorly-drafted CLA can itself be a liability); (c) it is the same mechanism used by the Linux kernel, Git, systemd, and many other high-profile projects, so it is well-understood; (d) it preserves the contributor's copyright (no assignment), which is what most contributors expect. The Linuxify context favors the DCO forces: the maintainer team is small and has no legal counsel on retainer, the project is MIT-licensed (so the patent-protection argument for a CLA is weaker than for a project with a custom license), and the contributor base is expected to be individual developers (not corporate employees) in the early years.

- Decision: Linuxify uses **DCO only, no CLA**. Every commit to every Linuxify repo (the monorepo per [ADR-010](adr-010-monorepo-vs-polyrepo.md), and any future split repos) must include a `Signed-off-by: Name <email>` line attesting to the DCO. The DCO text is included in `CONTRIBUTING.md` and is the same as the Linux kernel's. CI (a GitHub Actions workflow using `dco-bot` or equivalent) verifies that every commit in a PR has a `Signed-off-by` line and blocks the PR if not. The project does not require contributors to sign any separate agreement, does not use cla-assistant.io, and does not maintain a list of CLA signers. Contributors retain their copyright; the project receives a license under the MIT (or BSL for server components, per [ADR-014](adr-014-bsl-for-server-side-components.md)) terms. The decision is documented in `CONTRIBUTING.md` §"DCO" and is referenced from the [contribution guidelines](../16-community/contribution-guidelines.md).

- Consequences:
  - **Positive**: Contribution friction is minimal — `git commit -s` is one flag, and most developers who have contributed to the Linux kernel or Git already know the pattern. No legal review was needed to set this up; the DCO text is public domain boilerplate. Contributor copyright is preserved, which is what most contributors expect and what reduces the "I contributed to X and later they relicensed without asking me" complaint class. The DCO is sufficient IP protection for an MIT-licensed project: it certifies that the contributor had the right to contribute the code, which is the main thing a maintainer needs to defend against a future IP claim. The CI check is automatic and cheap.
  - **Negative**: A future relicensing (e.g., if Linuxify ever moved the CLI from MIT to a copyleft license) would require tracking down every contributor for permission, because the DCO does not grant relicensing rights. We accept this risk because (a) the CLI is MIT and we have no plans to relicense, (b) the server components are BSL (per ADR-014) from the start and do not need relicensing, and (c) if a relicense ever became necessary, the contributor count at that point would be small enough to track down. Some corporations have policies that prohibit employees from contributing to DCO-only projects without a CLA (because the DCO's patent grant is implicit rather than explicit); this may cost us some corporate contributions, but the v1 contributor base is expected to be individual developers. The DCO does not protect against a contributor pushing code they did not write (the DCO is an attestation, not a verification); we mitigate by requiring PR review and by running a periodic `git log` audit for commits whose `Signed-off-by` does not match the commit author.
  - **Neutral**: The DCO is the same mechanism used by the Linux kernel, Git, systemd, Docker, and many other projects, so it is well-precedented. A future switch to a CLA (if corporate contributions ever became important enough) would require a new ADR superseding this one.

- Alternatives Considered:
  - **CLA with copyright assignment (Apache-style)**: rejected because the maintainer team has no legal counsel, because copyright assignment discourages casual contributions, and because the project has no plans to relicense.
  - **CLA without copyright assignment (e.g., Apache CLA, or the Harmony CLA templates)**: rejected because the lower-friction DCO is sufficient for an MIT-licensed project, and because even a non-assignment CLA adds a signing step that suppresses casual contributions.
  - **No DCO, no CLA (trust only)**: rejected because it provides no IP protection at all — a contributor could push code they did not write, and the project would have no attestation to point to in defense.
  - **CLA for corporations, DCO for individuals (hybrid)**: rejected for v1 as too complex to administer; the v1 contributor base is expected to be individuals, and a hybrid can be added later if corporate contributions materialize.
