# ADR-013: Ed25519 for v2 package signing (over RSA or PGP)

- Status: proposed
- Date: 2025-02-05
- Deciders: Linuxify core team
- Context: Linuxify v1 uses a git-based registry (per [ADR-011](adr-011-git-based-registry-v1.md)) where package definitions are YAML files committed to a git repo, and integrity is provided by signed git commits (GPG or SSH). This is sufficient for v1 because the registry is metadata-only — the actual CLI binaries are downloaded from their original sources (npm, pip, cargo) and verified by those package managers' own integrity mechanisms. In v2, Linuxify will introduce **signed packages**: the registry will serve not just metadata but also signed manifests (and, optionally, signed binary tarballs for pre-built rootfs images and patched CLIs), so that a user can verify that a package was produced by a trusted maintainer and has not been tampered with in transit. This ADR concerns the signing algorithm for v2. Three options were seriously considered. **Ed25519** is an elliptic-curve signature scheme (Edwards-curve Digital Signature Algorithm over Curve25519) that produces 64-byte signatures, uses 32-byte public keys, signs in microseconds, and verifies in microseconds; it was designed by Daniel J. Bernstein et al. and is widely deployed (SSH, age, WireGuard, Signal, Minisign, cosign). **RSA** is the legacy public-key signature scheme; a secure RSA key today must be 2048 or 3072 bits (256 or 384 bytes), signatures are the same size as the key, signing is fast but verification is slow (10–100× slower than Ed25519). **PGP (OpenPGP)** is a suite (RSA or ECC keys, web-of-trust identity model, key servers, key signing parties) that has been the traditional open-source signing mechanism (Linux package signatures, kernel module signing); it is complex, has a high operational burden, and has well-documented usability problems (the "Web of Trust is dead" essays).

  The forces favoring Ed25519 are: (a) it is modern, fast, and small — a 64-byte signature embedded in a YAML manifest is negligible, while a 256-byte RSA signature is not; (b) it is widely supported in the Node ecosystem via `tweetnacl` (used by cosign, age, and many others) and via Node's built-in `crypto.sign` (Node 16+); (c) it has no parameter choices to get wrong (unlike RSA, where key size, padding scheme, and hash algorithm must all be chosen correctly and are routinely misconfigured); (d) it does not carry PGP's web-of-trust baggage — a public key is just a 32-byte string that the CLI pins in `~/.linuxify/trusted-keys/`. The forces favoring RSA are ecosystem familiarity (RSA is everywhere) and FIPS 140-2 compliance (Ed25519 was only added to FIPS 140-3 in 2023, and some enterprise environments still require RSA). The forces favoring PGP are existing infrastructure (Linux distributions already use it) and the web-of-trust identity model (which lets users verify that a signing key belongs to a specific human). The Linuxify context favors Ed25519: the v2 user base is on Android (where `gpg` is not installed by default), the packages are signed by a small set of maintainers (so web-of-trust is unnecessary — users pin the maintainer keys directly), and the FIPS compliance argument does not apply to a personal-device tool.

- Decision: Linuxify v2 uses **Ed25519** for package and manifest signing. Each maintainer generates an Ed25519 keypair (`linuxify signing keygen`), publishes the 32-byte public key to the registry (`keys/<maintainer>.pub`), and signs each package manifest with their private key, producing a 64-byte signature appended to the manifest as a `signature:` field. The CLI's `src/utils/crypto.ts` (see [AI Build Guide](../00-executive/ai-build-guide.md) §7) verifies signatures using the `tweetnacl` library (a small, audited, pure-JS Ed25519 implementation). The set of trusted maintainer public keys is pinned in the CLI source (so a compromised registry cannot add a new trusted key) and updated only via CLI releases. v1 will not implement signing (the git-commit signatures are sufficient); v2 will add it as documented in [package-registry-future](../19-future/package-registry-future.md) §3. The decision is documented here in advance so that v2 implementers do not relitigate the algorithm choice.

- Consequences:
  - **Positive**: Signatures are 64 bytes — negligible overhead in a YAML manifest. Verification is microseconds, so checking signatures on every `linuxify add` adds no perceptible latency. Key generation is milliseconds, so maintainers can rotate keys easily. The implementation is ~50 lines of TypeScript (`tweetnacl.sign.detached` and `tweetnacl.sign.detached.verify`), with no parameter choices to misconfigure. The "did the maintainer sign this?" question is a simple public-key lookup, not a web-of-trust traversal. Ed25519 is the same algorithm used by SSH (modern OpenSSH defaults), age, cosign, and Signal, so the format is familiar to security-conscious users.
  - **Negative**: Ed25519 is not FIPS 140-2 certified (it is in FIPS 140-3, but some older enterprise environments have not adopted 140-3); if a future enterprise user requires FIPS compliance, they would need to fork or use a different signing path. We accept this because the v1/v2 user base is individual developers on Android, not enterprises. `tweetnacl` is a third-party dependency (albeit small, audited, and widely used); an alternative is Node's built-in `crypto.sign('Ed25519', ...)`, which we will use in Node 16+ environments and fall back to `tweetnacl` in older environments. The pinned-trust model (CLI ships with a hardcoded list of maintainer public keys) means that adding a new maintainer requires a CLI release; we accept this because the maintainer set is small and changes rarely, and because it prevents a compromised registry from adding a new trusted key.
  - **Neutral**: The v1 git-commit signatures (GPG/SSH) remain in place for registry commits; the v2 Ed25519 signatures are an additional layer on top, applied to individual package manifests. The two mechanisms are complementary, not redundant. A future v3 might switch to a transparency-log model (like Sigstore's Rekor) for additional assurance; this is documented as a v3+ possibility.

- Alternatives Considered:
  - **RSA-3072 with PKCS#1 v1.5 padding**: rejected because the 384-byte signatures are larger, the verification is slower, and the parameter choices (key size, padding, hash) are routinely misconfigured. Reserved for a future FIPS-compliant enterprise variant if demand materializes.
  - **PGP / OpenPGP**: rejected because the operational burden (key servers, key signing parties, revocation certificates) is too high for a small maintainer team, because `gpg` is not installed on Android by default, and because the web-of-trust identity model is unnecessary when the trusted-key set is small and pinned.
  - **ECDSA over P-256**: rejected because Ed25519 has a simpler, harder-to-misuse API (no nonce to leak, no parameter choices) and because Ed25519 is more widely adopted in modern tooling.
  - **HMAC (symmetric)**: rejected because it requires a shared secret between signer and verifier, which does not fit the open-source model where any user can verify.
