# ADR-014: BSL for server-side components (over MIT or AGPL)

- Status: proposed
- Date: 2025-02-06
- Deciders: Linuxify core team
- Context: Linuxify v1 is entirely local — the CLI, the docs, the registry (a git repo per [ADR-011](adr-011-git-based-registry-v1.md)), and the example plugins are all MIT-licensed, and there is no server component. In v2, Linuxify will introduce two server-side components: a **sync server** (lets users sync their `~/.linuxify/` state, installed packages, and patch overrides across devices) and a **registry HTTP API server** (a thin HTTP layer over the git registry, per ADR-011's "v2 HTTP API" note). These components raise a licensing question that the v1 components did not: should the server code be MIT (like the rest of Linuxify), AGPL (copyleft for servers), or BSL (Business Source License, the "source available, eventually open" model used by MariaDB, Sentry, CockroachDB, HashiCorp, and others)?

  The forces favoring MIT for the servers are consistency (everything else is MIT) and maximum openness (anyone can host a sync server, including a competitor). The forces favoring AGPL are copyleft: anyone who operates a Linuxify sync server (and thus "conveys" the software under AGPL §13) must publish their modifications, which prevents a cloud provider (e.g., AWS) from taking the Linuxify sync server, wrapping it in a managed product, and contributing nothing back. The forces favoring BSL are sustainability and anti-competitiveness: BSL lets the source be readable and modifiable for non-competing use, but forbids competing production deployments for a fixed period (typically 4 years), after which the code converts to MIT/Apache. This is the model HashiCorp adopted for Terraform in 2023 (controversially) and that Sentry, MariaDB, and CockroachDB have used for years.

  The Linuxify context favors BSL for the server components specifically (not for the CLI, docs, or registry) for three reasons. First, the sync server is the natural "cloud" revenue surface — if Linuxify ever offers a hosted sync service (a paid product), MIT-licensing the server means a competitor can run the identical code at cost with no contribution back, which undercuts the sustainability of the project. Second, AGPL is bad UX for self-hosters: a user who self-hosts the sync server for personal use on their home network is technically "conveying" the software to their own devices, and AGPL's conveyance clause creates legal uncertainty (does syncing to my phone count as conveyance?) that discourages self-hosting. Third, BSL's "eventually MIT" clause (typically 4 years) means the code becomes fully open over time, which aligns with Linuxify's open-source values while preserving the sustainability window. The CLI, docs, registry, and plugins remain MIT because they are not server components and the sustainability argument does not apply.

- Decision: Linuxify's server-side components — the sync server (`server/sync/`) and the registry HTTP API server (`server/registry-http/`), both landing in v2 — are licensed under the **Business Source License (BSL) 1.1**, with a "change date" of 4 years from each component's first release and a "change license" of MIT. The BSL `LICENSE` file lives in each server subdirectory; the repo-root `LICENSE` remains MIT for the CLI, docs, registry, and plugins. The BSL's "Additional Use Grant" clause explicitly permits: (a) personal, non-commercial self-hosting by an individual; (b) internal use within an organization that does not operate a public Linuxify sync service; (c) any use that is not a "competing product" (defined as a hosted sync or registry service offered to third parties). After the 4-year change date, the code automatically becomes MIT. The decision is documented in `server/sync/README.md` and `server/registry-http/README.md`, and a `LICENSE-BSL` file in each server subdirectory spells out the terms. The CLI's `linuxify sync` command (per [cloud-sync](../19-future/cloud-sync.md)) supports both the official hosted service and a user-supplied self-hosted endpoint, so users are not locked in.

- Consequences:
  - **Positive**: Sustainability — the project can offer a hosted sync service without a competitor free-riding on the code, which funds ongoing development. Self-hosting is explicitly permitted for personal and internal use, so users who want to run their own sync server can. The 4-year change date means the server code becomes fully MIT over time, so the open-source commitment is concrete and time-bounded rather than indefinite. The license heterogeneity (MIT for most, BSL for servers) is handled at the directory level, so a user reading the repo sees `LICENSE` (MIT) at the root and `LICENSE-BSL` in `server/sync/`, with a clear `README.md` note in each. The BSL is well-understood by the open-source community (HashiCorp, Sentry, MariaDB, CockroachDB all use it), so potential contributors are not surprised.
  - **Negative**: BSL is not OSI-approved (it is "source available," not "open source" by the OSI definition), so some Linux distributions (notably Debian and Fedora) will not package the server components in their main repositories. We accept this because the server components are not user-installed packages (they are operated, not installed), and because the CLI (which is what distributions package) remains MIT. Some users will disagree with any non-OSI license and will refuse to use the hosted sync service; we accept this and provide a clean self-hosting path (which BSL explicitly permits) so users are not coerced. The "competing product" definition in the Additional Use Grant must be drafted carefully (by a lawyer, which the v1 maintainer team does not have) to avoid being either too narrow (letting a competitor exploit a loophole) or too broad (catching legitimate self-hosters); we mitigate by using the same Additional Use Grant text that Sentry and HashiCorp use, which has been vetted by their lawyers.
  - **Neutral**: The 4-year change date is the same as HashiCorp's and Sentry's, so it is industry-standard. A future ADR could shorten or lengthen the change date for specific components; this is a per-component decision, not a project-wide one.

- Alternatives Considered:
  - **MIT for the servers too**: rejected because it permits a competitor (AWS, etc.) to operate a managed Linuxify sync service without contributing back, which undercuts the sustainability of the project and the hosted service that funds development.
  - **AGPL for the servers**: rejected because AGPL's conveyance clause creates legal uncertainty for self-hosters (does syncing to my phone count as conveyance?) that discourages the self-hosting use case, and because AGPL's "must publish modifications" clause is hostile to enterprise internal use.
  - **Elastic License 2.0 (ELv2)**: rejected because it is less well-understood than BSL and because its "no removing license keys" clause is irrelevant to Linuxify (we have no license keys).
  - **Server Side Public License (SSPL)**: rejected because it is more aggressive than AGPL (requires publishing the entire stack, including non-Linuxify code, if you operate a service) and is widely viewed as hostile to community contribution.
