# ADR-015: Zod for runtime schema validation

- Status: accepted
- Date: 2025-01-24
- Deciders: Linuxify core team
- Context: Linuxify reads, parses, and acts on several classes of untrusted external input: package definition YAML files (from the registry, per [ADR-002](adr-002-yaml-package-definitions.md)), the user's `~/.linuxify/config.toml` (per [ADR-008](adr-008-toml-config-over-yaml-json.md)), plugin manifests (`plugin.toml` from third-party plugins, per [plugin-sdk](../10-plugin-sdk/plugin-sdk.md)), the `~/.linuxify/state.json` state file (which may have been corrupted by a previous crash), and the responses of the registry git repo (which could be tampered with by a compromised mirror). Every one of these inputs must be validated against a schema *before* any code acts on it, because acting on unvalidated input is the root of a large fraction of security bugs and confusing error messages. The question is which validation library to standardize on. Four options were seriously considered. **Zod** is a TypeScript-first schema library (no separate schema language, schemas are TypeScript objects, types are inferred via `z.infer<typeof schema>`) with first-class support for discriminated unions, refinements, transforms, and async validation; it is the most popular validation library in the modern TypeScript ecosystem (used by tRPC, shadcn/ui, Next.js, and many others). **Joi** is an older, JavaScript-focused validation library (originally part of the hapi ecosystem) with a fluent API; it predates TypeScript and its type inference is weaker. **ajv** is a JSON Schema validator (compile a JSON Schema draft 2020-12, get a validator); it is the most standards-compliant option (JSON Schema is an IETF draft) but is more verbose and the generated types are less ergonomic. **Hand-written validators** (a function per type that checks each field and returns a typed result) have zero dependencies and full control but are the most error-prone (every field must be checked, every type must be maintained by hand, and the validators drift from the types as the codebase evolves).

  The forces favoring Zod are: (a) TypeScript-native — the schema *is* the type definition, so there is one source of truth and the types cannot drift from the validators; (b) inferred types — `z.infer<typeof PackageSchema>` produces the TypeScript type, so a contributor who changes the schema gets the type update for free; (c) excellent error messages — Zod's default error format names the path (`patches[0].file`) and the expected type, which is what a user editing a YAML file needs to see; (d) discriminated unions — Linuxify's patcher has multiple patch types (regex, ast-js, ast-ts, sed, python-ast, shell) that Zod handles cleanly with `z.discriminatedUnion`; (e) ecosystem momentum — Zod is the de facto standard for new TypeScript projects in 2024–2025, so contributors are likely to already know it. The forces favoring ajv are standards compliance (JSON Schema is portable across languages, so the package-spec schema could be consumed by non-Node tooling). The forces favoring Joi are familiarity for contributors from the hapi ecosystem. The forces favoring hand-written are zero dependencies.

- Decision: Linuxify uses **Zod** for all runtime schema validation across the codebase. Every external input is parsed through a Zod schema before use: package definitions (`src/packages/schema.ts`), config (`src/config/schema.ts`), state (`src/state/schema.ts`), plugin manifests (`src/plugins/manifest.ts`), registry index (`src/registry/schema.ts`), telemetry events (`src/telemetry/events.ts`), and the patcher's patch-type union (`src/patcher/types.ts`). The TypeScript types for these structures are *always* `z.infer<typeof Schema>`, never hand-written interfaces, so the types cannot drift from the validators. For the package-definition schema specifically, a JSON Schema is *also* generated from the Zod schema (via `zod-to-json-schema`) and shipped at `docs/09-registry/package.schema.json`, so non-Node tooling (e.g., a future Rust-based linter, or a VS Code extension that does inline validation) can consume the schema without depending on Zod. This gives us Zod's ergonomics inside the CLI and JSON Schema's portability outside it. The package-definition YAML schema is therefore defined *once* in Zod and exported to JSON Schema at build time, avoiding the "two schemas that drift" problem.

- Consequences:
  - **Positive**: Single source of truth — the Zod schema is the type, the validator, and (via `zod-to-json-schema`) the JSON Schema, so there is no drift. Inferred types mean a contributor who adds a field to the schema gets the type update, the validator update, and the JSON Schema update for free. Error messages are user-friendly by default — a user who writes `runtime_min_verison: 20` (typo) in `cline.yml` gets `patches.0.file: expected string, got undefined` plus a path, not a cryptic "validation failed." Discriminated unions handle the patcher's six patch types cleanly. The contributor pool is large — Zod is well-known, so new contributors do not need to learn a bespoke validation framework.
  - **Negative**: Zod adds ~50 KB (minified) to the CLI's dependency tree, which is non-trivial for a CLI that aims for fast startup. We mitigate by lazy-loading Zod only in commands that need it (`linuxify add`, `linuxify config`); commands like `linuxify --version` never import it. Zod's runtime validation has a cost (~1–5 ms per schema parse for typical package-definition size), which is negligible compared to the cost of `npm install` but is non-zero; we mitigate by caching parsed schemas in memory. Some contributors from older Node ecosystems may be unfamiliar with Zod; we mitigate with a `docs/21-reference/zod-guide.md` (future) and with examples in the [plugin-sdk](../10-plugin-sdk/plugin-sdk.md) docs. Zod's API evolves (v3 → v4 changed several method names), so we pin a major version and document the migration in the changelog when we upgrade.
  - **Neutral**: The JSON Schema export means a future Rust-based Linuxify (or any non-Node tool) can validate package definitions using the same schema, so the validation logic is portable across implementations. The hand-written-validator alternative would have made this portability impossible. The decision is reversible (a future ADR could switch to ajv, generating Zod-equivalent types via `json-schema-to-typescript`), but the ergonomics would be worse and no contributor has requested it.

- Alternatives Considered:
  - **Joi**: rejected because its TypeScript type inference is weaker than Zod's, because its API predates TypeScript and feels anachronistic in a modern codebase, and because its ecosystem momentum has shifted to Zod.
  - **ajv (JSON Schema)**: rejected as the *primary* validation library because hand-authoring JSON Schema is more verbose than Zod and the inferred types are less ergonomic; ajv is used *indirectly* via the JSON Schema export for non-Node tooling.
  - **Hand-written validators**: rejected because they drift from the types (every field must be updated in two places), because they are error-prone (every field must be checked), and because the maintenance burden grows quadratically with the number of fields.
  - **io-ts**: rejected because its API is more functional-programming-flavored (less approachable for contributors) and because its ecosystem momentum has been eclipsed by Zod.
  - **runtypes**: rejected for similar reasons to io-ts; Zod has won the ecosystem.
